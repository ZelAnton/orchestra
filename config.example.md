# Пример `.work/config.md`

Скопируйте нужные ключи в `.work/config.md` вашего проекта. Отсутствующий файл или
отсутствующий ключ означает значение по умолчанию (сводка дефолтов — в таблице
ниже) — необязательно копировать все ключи, только те, что хотите переопределить.

**Исключение — `CODEX_CODER` и `CODEX_REVIEWER`** читаются с фолбэком на переменные
окружения ОС: `config.md` → переменная окружения (`CODEX_CODER` / `CODEX_REVIEWER`)
→ дефолт `off`. Если файла нет или ключ в нём не задан, но выставлена одноимённая
переменная окружения — берётся её значение; ключ в `config.md` всегда переопределяет
окружение. Удобно включить codex глобально (в профиле/CI) без правки `config.md`
каждого проекта. Остальные ключи — только из `config.md`.

`launchers/cc-config.cmd` сеет `.work/config.md` из блока ниже, ограниченного
маркерами `# >>> config.md seed start >>>` / `# <<< config.md seed end <<<`, —
копируются только сами маркеры-границы и всё между ними, без заголовков и
пояснительного текста этого документа.

```
# >>> config.md seed start (cc-config.cmd copies only this block) >>>
MAX_PARALLEL: 5
# --- Роллинг-приём когорты (см. ниже) ---
# COHORT_SIZE: 15        # по умолч. 3×MAX_PARALLEL — сколько задач максимум принять
                          #   в одну когорту, прежде чем закрыть приём
# COHORT_MAX_AGE: 90      # по умолч. 90 (минут) — сколько держать приём открытым по
                          #   возрасту когорты; закрывается по тому, что наступит раньше
REVIEW_MIN_PASSES: 2
REVIEW_LOOP_MAX: 8
INTEGRATION_LOOP_MAX: 8
CI_FIX_MAX: 5
QUARANTINE_MAX_ATTEMPTS: 3
# --- Supervisor выполнения (дедлайн/попытки/объём вывода/бюджет; см. ниже) ---
# CALL_DEADLINE_SEC: 1800        # дедлайн одного вызова исполнителя (сек): превышение →
                                 #   таймаут (дерево процессов снимается, безопасный ретрай)
# CALL_MAX_ATTEMPTS: 2           # сколько раз supervise повторяет вызов при ТРАНЗИЕНТНОМ
                                 #   сбое (таймаут/крэш), прежде чем эскалировать
# CALL_OUTPUT_MAX_BYTES: 1048576 # предел объёма захватываемого вывода вызова (байт)
# COHORT_BUDGET_SEC: 0           # общий бюджет стенных часов когорты (сек); 0 = без лимита
# SMOKE_CMD: npm test   # пример; по умолчанию smoke не запускается
PUSH: true
CI_WATCH: true
# --- Гейт публикации: ожидание обязательных CI-проверок (Фаза 5.4; см. ниже) ---
# PUBLISH_CI_DEADLINE_SEC: 1800  # сколько ждать полного набора обязательных CI-проверок,
                                 #   прежде чем fail-closed (не публиковать/не архивировать)
# PUBLISH_CI_BACKOFF_SEC: 30     # пауза между опросами CI в цикле ожидания
# APPROVAL_DEADLINE_SEC: 86400   # срок действия запроса на human approval; нет ответа к
                                 #   дедлайну = отказ (fail-closed), не одобрение по умолчанию
REVIEWER_TIERING: true
# MAIN_BRANCH: main      # по умолчанию — автоопределение (см. ниже)
# --- Событийный outbox (.work/events.jsonl) ---
# EVENTS_OUTBOX: off     # on (по умолч.) | off — писать ли машинные события в
                         #   .work/events.jsonl (append-only машинный контракт для
                         #   будущей платформы наблюдаемости; см. ниже)
# --- База знаний проекта (.work/knowledge/) ---
# KB: on                 # off (по умолч.) | on — чтение KB + Фаза 5.5 (knowledge_curator)
# KB_TTL: 8              # истечение неподтверждённых одиночных записей, в батчах
# KB_CAP: 12             # максимум записей на область
# --- Codex-агенты (опционально; требуют установленного codex + `codex login`) ---
# CODEX_CODER: fast+std        # реализация/R--фиксы: off (по умолч.) | fast | fast+std
# CODEX_REVIEWER: fast+std      # ревью: off (по умолч.) | fast | fast+std | deep
                               # ^ РЕКОМЕНДУЕТСЯ при наличии codex: снимает самый дорогой
                               #   повторяющийся расход (opus reviewer) и ускоряет loop;
                               #   без codex — чистый фолбэк на Claude (лишний спавн на задачу).
# CODEX_CIFIX: on              # точечные CI/сборочные фиксы (Режим 3): off (по умолч.) | on
# CODEX_MODEL: gpt-5-codex     # пусто → модель из ~/.codex/config.toml
# CODEX_REASONING: auto        # auto | low | medium | high
# CODEX_SANDBOX: workspace-write
# CODEX_NETWORK: on            # сеть в песочнице coder_codex: on (по умолч.) | off
# CODEX_CMD: codex
# <<< config.md seed end <<<
```

## Что означает каждый ключ

- `MAX_PARALLEL` — сколько задач максимум исполняется **одновременно** в когорте.
  Processor держит этот пул полным роллингом (см. «Роллинг-приём когорты» ниже) — как
  только слот освобождается, тут же добирается следующая подходящая задача, не
  дожидаясь завершения всей когорты. Для очередей из мелких независимых задач
  (типичный вывод `code_auditor`/`enhancement_scout`) можно смело держать на 5 и выше.
  Для тяжёлых/взаимозависимых задач имеет смысл понижать. Ограничитель — способность
  planner находить столько непересекающихся доменов одновременно.
- `COHORT_SIZE` / `COHORT_MAX_AGE` — см. «Роллинг-приём когорты» ниже.
- `REVIEW_MIN_PASSES` — минимальное число прогонов поиска проблем у reviewer/
  full_reviewer на **первом** ревью задачи/батча (цикл 1). На повторных циклах
  (2+) действует режим повторного ревью (см. reviewer.md/full_reviewer.md) —
  этот параметр их не увеличивает.
- `REVIEW_LOOP_MAX` — сколько раз processor повторяет цикл «reviewer →
  исполнитель чинит находки» по одной задаче, прежде чем эскалировать её.
- `INTEGRATION_LOOP_MAX` — то же самое для цикла full_reviewer на джойн-барьере.
- `CI_FIX_MAX` — сколько раз processor пытается «упал CI → фикс → push», прежде
  чем остановиться и запросить ручное вмешательство.
- `QUARANTINE_MAX_ATTEMPTS` — предел числа **попыток** задачи (счётчик `попытка=N`
  в суффиксе строки очереди): при `N ≥ QUARANTINE_MAX_ATTEMPTS` задача помечается
  терминально `эскалирована` вместо бесконечного повторного захвата. При дефолте 3
  это 3 попытки (то есть 2 возврата в очередь после карантина неразрешимого
  конфликта слияния).
- `CALL_DEADLINE_SEC` — дедлайн одного **вызова исполнителя** (coder/reviewer/merger/
  codex), в секундах. Превышение классифицируется как `timeout`: supervisor снимает всё
  дочернее дерево процессов и это транзиентный сбой (безопасный ретрай). Раздел
  «Supervisor выполнения» ниже.
- `CALL_MAX_ATTEMPTS` — сколько раз supervisor повторяет один вызов при **транзиентном**
  сбое (`timeout`/`crash`), прежде чем эскалировать; **не** путать с
  `QUARANTINE_MAX_ATTEMPTS` (пределом попыток задачи в очереди). Содержательная ошибка
  исполнителя (`error`) не ретраится вовсе — идёт в quarantine/requeue.
- `CALL_OUTPUT_MAX_BYTES` — предел объёма захватываемого вывода одного вызова, в байтах:
  свыше него транзиентный захват усекается (в долговременную запись сырой вывод не
  попадает вовсе — см. приватность в разделе ниже).
- `COHORT_BUDGET_SEC` — общий бюджет стенных часов на всю когорту, в секундах; `0` = без
  лимита. При исчерпании бюджета supervisor не запускает следующий вызов (`budget`), а
  дедлайн отдельного вызова подрезается до остатка бюджета.
- `SMOKE_CMD` — команда быстрой проверки сборки/тестов, которую использует
  merger после каждого слияния и coder — при самопроверке. Если не задано,
  агенты сами решают, есть ли в проекте подходящая команда (`npm test`,
  `pytest -q`, `go build ./...` и т.п.), и пропускают проверку, если не нашли.
  **Предвыдача согласия для `SMOKE_CMD` — пер-репозиторно, не централизованно.**
  `SMOKE_CMD` — произвольная команда конкретного проекта, поэтому в **центральный**
  (общий для всех репозиториев, сеемый `cc-config`) канонический allow-список её
  **не включают**: это неявно раздуло бы полномочия во всех проектах сразу. Если в
  permission-mode `auto` classifier Claude Code блокирует вашу `SMOKE_CMD` посреди
  прогона, предвыдайте согласие **пер-репозиторно** тем же механизмом, что и для codex
  (T-058): добавьте локальное allow-правило под свою команду в
  `.claude/settings.local.json` этого репозитория (например `Bash(npm test:*)`,
  `Bash(pytest:*)`) — руками или через `/permissions` в интерактивной сессии. Это
  осознанное решение оператора **данного** репозитория, а не часть общего дистрибутива;
  `cc-config` такое правило сам не сеет (сеет только канонические codex-правила
  `Bash(pwsh -File tools/codex-runtime.ps1 *)` и `Bash(codex exec *)`, см. «Разрешение на
  запуск codex» ниже). Отказ в разрешении посреди прогона
  обрабатывается штатной эскалацией роли (coder/merger возвращают блокер в отчёт), а не
  самопредоставлением прав — единую политику см. в `knowledge.md`, «Разрешения auto-режима
  и политика "согласие — заранее"».
- `PUSH` — `false` отключает push и ожидание CI на джойн-барьере (например, для
  локального прогона без remote), но processor всё равно делает ff-merge в основную
  ветку и на этом останавливается — слитый код остаётся в main, просто не
  публикуется.
- `CI_WATCH` — `false` пропускает ожидание CI даже если push выполнен (полезно,
  если CI настроен, но не хочется ждать в автоматическом прогоне).
- `PUBLISH_CI_DEADLINE_SEC` — сколько секунд processor ждёт полного набора обязательных
  CI-проверок публикации (Фаза 5.4), прежде чем считать ожидание истёкшим и **не**
  публиковать/архивировать (fail-closed). Набор обязательных проверок задаётся политикой
  `.work/constraints.md` (раздел «Обязательные CI-проверки публикации»); дедлайн — только
  тайминг ожидания. По умолчанию 1800 (30 минут).
- `PUBLISH_CI_BACKOFF_SEC` — пауза между опросами CI-проверок в цикле ожидания Фазы 5.4.
  По умолчанию 30 секунд.
- `APPROVAL_DEADLINE_SEC` — срок действия запроса на человеческое подтверждение
  (`tools/policy.ps1 approval-request`: обязательный human review, force-lock, policy
  bypass). Нет ответа оператора к дедлайну = **отказ** (fail-closed), не одобрение по
  умолчанию. По умолчанию 86400 (24 часа).
- `REVIEWER_TIERING` — `false` отключает тиринг ревью (см. reviewer_std.md) и
  всегда использует полный `reviewer` (opus/high) независимо от уровня
  исполнителя задачи. Включено по умолчанию — экономит существенную часть
  затрат на очередях из простых задач.
- `MAIN_BRANCH` — имя магистральной ветки (trunk), в которую processor ff-мержит
  и пушит. По умолчанию — **автоопределение**: в git — `origin/HEAD` → `main` →
  `master`; в jj — `main` → `master`. Задавайте явно для репозиториев, чей trunk
  называется иначе (`develop`, `trunk` и т.п.), где автоопределение промахнётся.
- `EVENTS_OUTBOX` — `on` (по умолчанию) заставляет processor дописывать машинные
  события в append-only `.work/events.jsonl` (одна JSON-строка на событие)
  параллельно с человекочитаемыми Markdown-артефактами; `off` — не писать вовсе
  (поведение как раньше). Подробнее — раздел «Событийный outbox» ниже.

## Роллинг-приём когорты

Вместо того чтобы набирать фиксированный батч раз и ждать, пока весь он дойдёт до
джойн-барьера, processor **непрерывно доукомплектовывает** когорту: как только слот
освобождается (задача дошла до `готова к слиянию` или эскалирована) и приём ещё
открыт — тут же добирается следующая подходящая задача из очереди (риск конфликта
доменов проверяет `planner` против всех текущих активных). Это устраняет простой
слотов, пока одна медленная задача (много циклов ревью/фиксов) держит батч открытым.

Приём закрывается по тому, что наступит раньше:
- `COHORT_SIZE` — принято достаточно задач (по умолчанию `3×MAX_PARALLEL`: втрое
  больше пула одновременного исполнения — даёт роллингу простор, не раздувая
  join-барьер запредельно);
- `COHORT_MAX_AGE` — прошло достаточно времени с начала когорты (по умолчанию 90
  минут) — держит `main` от слишком долгого простоя без публикации;
- очередь `не начата` действительно опустела — закрывает приём независимо от обоих
  лимитов выше.

После закрытия приёма новые задачи в эту когорту не берутся — активные доводятся до
терминального состояния, затем джойн-барьер (интеграционное ревью + ff-merge в main +
push + CI) отрабатывает как обычно, один раз на всю накопленную когорту.

**Компромисс, который стоит понимать:** `merger` сливает ветки последовательно, а
`full_reviewer` сканирует межзадачную поверхность по числу слитых веток — чем больше
`COHORT_SIZE`, тем длиннее сам join-барьер (это меньшая, но всё же ограниченная версия
той же проблемы хвостовой задержки, только перенесённая с приёма на слияние).
Настраивайте оба ключа по факту прогонов — `journal.md` пишет число волн приёма и
причину закрытия для каждой когорты.

## Supervisor выполнения

`tools/supervisor.ps1` — исполняемый ограничитель одного **вызова исполнителя**
(coder/reviewer/merger/codex). Он не трогает очередь, аренду и outbox — он оборачивает
ровно один вызов и возвращает processor неконфиденциальный вердикт, по которому тот
маршрутизирует дальнейшую обработку. Четыре причины остановки, каждая со своим кодом
возврата и своим путём обработки:

- **`ok`** (0) — вызов завершился успешно.
- **`timeout`** (3) — превышен `CALL_DEADLINE_SEC`; всё дочернее дерево процессов снято.
  **Транзиентно**: безопасный ретрай (`supervise` повторяет до `CALL_MAX_ATTEMPTS`, затем
  эскалирует).
- **`cancelled`** (4) — запрошена кооперативная отмена (появился cancel-файл — например
  kill-switch `.work/PAUSE` — или оператор остановил прогон). Дерево снимается, пишется
  resume-**checkpoint**; это не сбой: слот освобождается, независимые готовые задачи
  продолжают прогресс.
- **`crash`** (5) — крэш инструмента/инфраструктуры: процесс не удалось породить, он убит
  сигналом или вышел с кодом из `--crash-exit-codes`. Транзиентно, как `timeout` →
  ограниченный ретрай.
- **`error`** (6) — **содержательная** ошибка исполнителя (вызов отработал и вернул
  content-level провал). Это путь quarantine/requeue (ограничен `QUARANTINE_MAX_ATTEMPTS`),
  **не** бесконечный ретрай.
- **`budget`** (7) — бюджет когорты (`COHORT_BUDGET_SEC`) исчерпан; вызов не запускается.

**Checkpoint** (`.work/tasks/<T-ID>/supervisor_checkpoint.json`) на отмене/таймауте
**совместим** с арендой Фазы 0 (`orchestrator.lock/lease.json`): несёт `owner_id`,
heartbeat и ttl, фиксирует задачу/батч/попытку/причину/затраченное время и **никогда** не
удаляет и не переписывает саму аренду — resume перечитывает его и продолжает ту же задачу
под тем же владельцем.

**Приватность/наблюдаемость.** Долговременный вердикт (`--result-file`) и события
`codex.attempt`, которые эмитит `observe`, несут только **скаляры** — причину, код выхода,
`duration_ms`, попытку, число байт и sha256 вывода, плюс классификаторную `outcome_reason`
(«deadline exceeded (Ns)», «exit code N», …), но **никогда** сырой вывод вызова. Сырой
stdout/stderr попадает только в транзиентный захват (усечённый до `CALL_OUTPUT_MAX_BYTES`),
не в outbox/journal, так что чувствительный вывод не утекает (совместимо с
`tools/redaction.ps1`, через который `observe` дополнительно прогоняет свободный текст).
`status.md`/`journal.md`/`events.jsonl` показывают elapsed, попытки, остаток бюджета и
причину остановки.

Матрица устойчивости этого пути — `tools/harness.ps1` (полный жизненный цикл когорты для
git и jj с fault injection после каждого критического перехода): быстрый срез
(`tests/test-harness.ps1`) и supervisor-тесты (`tests/test-supervisor.ps1`) — обязательный
шаг CI; расширенная crash-матрица (`tests/test-harness-crashmatrix.ps1`) — отдельная,
время-лимитированная, не блокирует обычный прогон.

## Пауза конвейера — kill switch `.work/PAUSE`

`.work/PAUSE` — не ключ конфига, а файловый **kill switch** для штатной остановки
работающего processor (в отличие от обрыва процесса). Пока файл существует, processor не
начинает новую мутирующую фазу/раунд: на ближайшей границе он оставляет согласованное
состояние, освобождает `.work/orchestrator.lock` и завершается — без брошенных на
середине worktree/merge. Проверяется только **наличие** файла; его содержимое (метка
времени/причина, которую можно передать аргументом) — информационное.

- **Поставить паузу:** `launchers/cc-pause` (`.cmd`/`.sh`), опционально с причиной:
  `cc-pause "reason"` — создаёт `.work/PAUSE`.
- **Снять паузу:** `launchers/cc-unpause` (`.cmd`/`.sh`) — удаляет `.work/PAUSE`.
- **Продолжить:** после снятия файла запустите `cc-processor` (или `cc-resume` для живой
  сессии) — processor продолжит с места остановки штатной recovery-логикой (Фаза 0), без
  ручной правки состояния.

Это **не** `cc-resume`: тот возобновляет прерванную сессию, а `cc-pause`/`cc-unpause`
поднимают/снимают намеренный запрос на остановку. Файл лежит в `.work/` (gitignored) — в
VCS не попадает; запускайте лаунчеры из корня проекта (папки, где лежит `.work/`).

## Политика ограничений проекта — `.work/constraints.md`

`.work/constraints.md` — не ключ конфига, а отдельный **человекочитаемый policy-файл**
проекта рядом с `.work/config.md`: запрещённые пути (denylist), разрешённые ветки/remotes,
push/merge policy, обязательные проверки, пороги размера изменений и категории обязательного
human review. Роли читают его, чтобы ограничить радиус изменений (blast radius): processor —
перед захватом когорты и повторно перед публикацией; planner — при нарезке задач;
coder/reviewer — при реализации и проверке границ diff.

Шаблон — корневой `constraints.example.md`; `launchers/cc-config` сеет из него
`.work/constraints.md` при первом запуске (**весь файл целиком**, в отличие от блочного
seed `config.md`), никогда не перезаписывая существующий. Значения по умолчанию минимальны
и безопасны для незнакомого проекта (пустой denylist, а не запрет всего). **Нет файла —
деградация без ошибок**: поведение как раньше, никаких новых жёстких отказов. Файл лежит в
`.work/` (gitignored), под контроль версий не попадает.

Раздел **категории обязательного human review** этого файла — источник (происхождение)
категорий для базовой risk-классификации задач: planner по ним присваивает полю
`Риск: low | medium | high` дескриптора `.work/tasks/<T-ID>/task.md` значение (нет файла —
оценивает риск сам, без ошибок), coder не вправе понижать заявленный риск, а reviewer
сверяет его с фактически затронутыми путями и может лишь повысить. Отдельного ключа
конфигурации для этого не вводится — категории живут в `.work/constraints.md`, а не в
`config.md`. Полную же модель human gates (парковку задач до решения человека) и
supervised-режим по риску этот файл пока не реализует — только базовые ограничения
(denylist, push/merge policy, обязательные проверки, пороги, пометка human-review
категорий) и происхождение категорий риска. См. `plans/LOOP_ORCHESTRA_ROADMAP.md`.

**Единый источник и исполняемая граница (T-084).** И этот `config.md`, и политику
`constraints.md` описывает один versioned schema source — `tools/policy-schema.ps1` (типы,
defaults, enum/range, env-precedence, чувствительность, разделы политики). Его исполняет CLI
`tools/policy.ps1` (companion `state-tx.ps1`/`queue-tx.ps1`): `validate-config` (fail-closed:
неизвестный/дублирующийся/невалидный ключ — ошибка, а не тихий default), `validate-policy`,
`migrate` (перенос старого `config.md` на текущую схему без потери значений/комментариев),
`guard-path` (гард destructive-операций: канонизация пути, корень/объект/leaf, id
задачи/батча, VCS-связь, защита от `..`/symlink/junction/подмены worktree), `check-paths`
(фактические изменённые пути против denylist после каждого возврата исполнителя и перед
commit/merge/publication) и `check-publish` (allowed branch/remote + push/merge policy как
технический precondition). Список ключей и множества значений Codex-ключей в этом файле,
allow-список `cc-doctor` и валидация processor'а — **машинно-сверяемые копии** этой схемы
(`tools/check-consistency.ps1`, Class 4/5), а не независимые источники: расхождение валит
smoke-гейт. `cc-doctor` держит свою копию хардкодом намеренно — он работает и вне репозитория
(mirror в `~/.claude/scripts`, где `tools/` недоступен), — но разойтись со схемой не может.

## База знаний проекта (KB)

`KB: on` включает единую базу знаний в `.work/knowledge/` (шарды `architecture/`/
`conventions/`/`pitfalls/` + `INDEX.md`), чтобы агенты **меньше исследовали код** и **не
повторяли ошибки**. По умолчанию `off` — тогда ничего не активируется, поведение как
раньше; при отсутствии каталога всё деградирует без ошибок.

- **Читают (pull-модель):** planner кладёт в дескриптор лишь короткие релевантные ловушки
  + указатели областей; coder/reviewer, узнав свои файлы, сами подтягивают записи из
  `INDEX.md` по scope. **Асимметрия доверия:** `pitfalls` можно доверять (худшее — лишняя
  проверка); `architecture`/`convention` (механизм) — перепроверять по коду, код побеждает.
- **Пишет только `knowledge_curator`** — на джойн-барьере (Фаза 5.5, после того как CI
  устаканился, до чистки Фазы 6). Харвестит дистиллят батча из **существующих** артефактов
  (находки `R-`/`F-`, `quarantined=`, причины падений CI и их фиксы, эскалации, опц.
  `learnings.md`) — нового груза на исполнителей нет. Идемпотентен по `B-id`; спавнится
  **условно** (только когда есть что харвестить) — на коротких/чистых батчах не тратится.
- `KB_TTL` — за сколько нетронутых батчей истекает неподтверждённая одиночная запись.
- `KB_CAP` — максимум записей на область (лишнее куратор сливает/выбрасывает по
  `confidence×recency`).

Инвариант «один писатель» держится на одном `orchestrator.lock`. Каталог машинно-локальный
(gitignored в `.work/`) — на нескольких клонах/машинах базы расходятся; это осознанный
компромисс v1 (durable-промоушен `architecture`/`conventions` в коммитимый каталог — на
будущее).

## Событийный outbox — `.work/events.jsonl`

`EVENTS_OUTBOX: on` (по умолчанию) включает **машинный event-outbox**: processor дописывает
одну строку валидного JSON на событие в append-only `.work/events.jsonl` — параллельно с
человекочитаемыми Markdown-артефактами (`status.md`, `journal.md`, дескрипторы задач),
которые **остаются источником истины для человека**. Файл — стабильный машинный контракт
для будущей платформы наблюдаемости (`plans/OBSERVABILITY_PLATFORM_PLAN.md`): по нему
дашборд/runner строит проекции состояния без парсинга Markdown. `off` — события не пишутся,
поведение как раньше.

- **Что записывается:** границы фаз и роллинг-раундов когорты, захват задачи, каждая смена
  статуса дескриптора (`в работе → на ревью → готова к слиянию → слита → опубликована →
  выполнена`), карантин и эскалация, слияние и публикация. Envelope события (поля
  `schema_version`/`event_id`/`occurred_at`/`type`/`batch_id`/`task_id`/`actor`/`payload`)
  и полный набор типов — в `agents/processor.md`, раздел «Событийный outbox».
- **Кто пишет:** только processor (инвариант «один писатель» держится на одном
  `orchestrator.lock`); листовые агенты в него не пишут.
- **Долговечность:** outbox **не** привязан к одной когорте — копит события всех когорт
  подряд, **не** удаляется и **не** усекается очисткой Фазы 6, никогда не переписывается
  задним числом. Будущий runner читает его по собственному курсору; дедупликация — по
  `event_id`.
- **Аддитивность:** запись событий строго дополняет Markdown-артефакты и не влияет на
  control-flow, гейты и crash-recovery processor; сбой дозаписи не роняет прогон.
- Файл лежит в `.work/` (gitignored, machine-local) — под контроль версий не попадает; на
  разных клонах/машинах наборы событий локальны (как и `.work/knowledge/`).

## Чувствительные данные: trust-контракт и redaction

Внешние данные (тела issue/PR, пользовательский источник очереди, внешние логи, вывод
сторонних инструментов, CI-логи) и чувствительные значения (токены, ключи, URL-credentials,
заголовки авторизации, PII) проходят **единый безопасный путь** — не как отдельный ключ
`config.md`, а как обязательный контракт всех ролей. Нормативное описание —
`docs/queue_contract.md`, §18; исполняемая половина — `tools/redaction.ps1` (детерминированный,
офлайновый, работает и под Windows PowerShell 5.1).

- **Trust/provenance.** У каждого входа есть происхождение и уровень доверия
  (`trusted`/`internal`/`external`); всё внешнее по умолчанию `external` и трактуется как
  **данные, а не инструкции** — оно не может изменить полномочия, маршрут исполнения или
  правила работы принимающей роли. Дословная внешняя цитата вносится только **ограниченным
  data-блоком** (`tools/redaction.ps1 wrap`): провенанс-заголовок + тело, где каждая строка
  экранирована так, что prompt-injection не превращается в управляющий текст; URL не
  открывается автоматически (defang `http`→`hxxp`).
- **Redaction перед записью.** До записи любого артефакта (`status.md`, `journal.md`,
  `events.jsonl`, дескрипторы, `knowledge/*`, тела записей очереди) и до передачи следующей
  роли текст проходит `tools/redaction.ps1 redact`: чувствительные значения заменяются на
  необратимый маркер `[redacted:<категория>:<fp8>]` со стабильным fingerprint (для отладки),
  но без возможности восстановить значение. Pipeline идемпотентен и **не применяется к
  исходному коду/diff**. Набор детекторов высокоточный — обычный текст (git-SHA, URL без
  credentials) не задевается, поэтому читаемость `status.md`/`journal.md` не страдает.
- **Project-specific паттерны.** Дополнительные регэкспы редактирования проект задаёт в
  `.work/constraints.md`, раздел `## Redaction patterns` (по одному regex на пункт `-`). Нет
  файла или раздела — деградация без ошибок (применяется только базовый набор).
- **Полный нередактированный вывод — только под human gate.** Полный чувствительный текст
  допускается **лишь** в явно настроенном локальном защищённом режиме; у `tools/redaction.ps1`
  **нет** собственного bypass-переключателя. Любой обход redaction требует **human gate
  T-095** — точка интеграции описана как расширение (`docs/queue_contract.md`, §18.5), сам
  T-095 пока не реализован, поэтому пути без redaction сейчас нет. Ключ конфигурации для
  защищённого режима вводится будущим T-095 через этот gate, а не текущим поведением.

## Codex-агенты (coder_codex, reviewer_codex)

Необязательные адаптеры поверх OpenAI **Codex CLI**: часть работы вместо Claude выполняет
codex, экономя квоту Claude для самых сложных задач (`coder_deep`, тяжёлые ревью). Все три
ключа по умолчанию выключены; при недоступности/сбое codex шаг **автоматически
откатывается** на эквивалентного Claude-агента — батч не страдает.

### Допустимые значения Codex-ключей (валидация, fail-closed)

Шесть Codex-ключей имеют строго ограниченные множества значений. Невалидное значение —
это **ошибка конфигурации**, а не повод молча подставить default: и диагностика
(`cc-doctor` — его движок `tools/doctor-runtime.ps1`), и runtime-путь (processor перед
маршрутизацией — Фаза 1.1; адаптеры `coder_codex`/`reviewer_codex` при формировании
`--sandbox`) валидируют значения по **этой одной таблице** — она единый источник истины.
`cc-doctor` и processor держат её копию (mirror-совместимость — движок `cc-doctor`
работает и вне репозитория, будучи зеркалирован рядом с launcher'ами в
`~/.claude/scripts`), а `tools/check-codex-config-guard.ps1` **машинно стережёт**, что
копии не разошлись между собой (defaults, множества значений, ветвление `processor.md` и
единый движок `cc-doctor` `tools/doctor-runtime.ps1`).

| Ключ | Допустимые значения | По умолчанию |
|---|---|---|
| `CODEX_CODER` | `off` \| `fast` \| `fast+std` | `off` |
| `CODEX_REVIEWER` | `off` \| `fast` \| `fast+std` \| `deep` | `off` |
| `CODEX_CIFIX` | `off` \| `on` | `off` |
| `CODEX_REASONING` | `auto` \| `low` \| `medium` \| `high` | `auto` |
| `CODEX_SANDBOX` | `read-only` \| `workspace-write` | `workspace-write` |
| `CODEX_NETWORK` | `on` \| `off` | `on` |

Правила валидации:
- **Пустой ключ → его default** (для `CODEX_CODER`/`CODEX_REVIEWER` — после фолбэка на
  одноимённую переменную окружения ОС; env-значение проходит **ту же** валидацию, что и из
  файла). Пустое/неустановленное — не ошибка.
- **Непустое значение вне множества** (опечатка, неверный вариант) → запуск когорты
  останавливается **до захвата задач** (Фаза 1/1.1) с указанием **ключа**, **фактического
  (некорректного) значения** и **списка допустимых** — без молчаливой подмены default.
- `CODEX_SANDBOX` намеренно **исключает `danger-full-access`** и любое иное значение,
  расширяющее запись за пределы рабочей копии задачи (совпадает с «валидированной формой»
  выше: `--sandbox read-only|workspace-write`). `reviewer_codex` при этом **всегда**
  принудительно read-only независимо от значения ключа (ревью не мутирует код).
- `CODEX_MODEL` и `CODEX_CMD` — свободнозначные строки (имя модели / бинаря), множеством
  не ограничены и в этой валидации не участвуют.

**Предпосылки:** установленный `codex` и разовый `codex login` (codex хранит креды в
`~/.codex/auth.json` — оркестр ключей не передаёт); опционально `AGENTS.md` в корне
репозитория, дублирующий запрет коммитов/мутаций VCS. Проверить готовность —
`launchers\cc-doctor.cmd`.

**Разрешение на запуск codex (предвыданный грант).** В permission-mode `auto` classifier
Claude Code отклоняет автономный запуск codex как «запуск автономного агента», причём отказ
приходит **посреди** прогона. С задачи T-075 адаптеры (`coder_codex`/`reviewer_codex`) гонят
codex **не** голой командой `codex exec`, а через runtime-обёртку: реально исполняемая
Bash-строка — это `pwsh -File tools/codex-runtime.ps1 <run|guard-commit|cleanup|…>` (сам
`codex exec` обёртка порождает дочерним процессом, .NET ProcessStartInfo, мимо
permission-гейта Bash). Чтобы codex запускался без единого вопроса, launcher'ы
`cc-processor`/`cc-resume` (`.cmd` и `.sh`) передают сессии **два** гранта `--allowedTools`:
`Bash(pwsh -File tools/codex-runtime.ps1:*)` (покрывает фактическую команду обёртки — один
префикс на все её подкоманды) и `Bash(codex exec:*)` (исторический якорь) — сам запуск
launcher'а пользователем и есть выдача согласия. Explicit allow-правило проверяется
permission-движком Claude Code **раньше** classifier'а, поэтому команда разрешается без
запроса и без отказа. Правила **префиксные** (матч по литеральному началу строки команды).
**Нестандартный `CODEX_CMD`** (не `codex`) отдельного правила **не** требует: он передаётся
обёртке аргументом `--codex-cmd`, поэтому реально исполняемая Bash-строка остаётся той же
runtime-обёрткой, которую грант уже покрывает. Адаптерам **запрещено** править
`.claude/settings*.json` для самопредоставления прав — согласие выдаётся заранее, через
launcher. Ad-hoc сессии `claude`, запущенные **не** через эти launcher'ы, сессионного гранта
не получают; для них (и чтобы устойчиво удовлетворять пер-сессионный permission-гейт
processor'а) `cc-config` обеспечивает **постоянный** allow-список в
`.claude/settings.local.json` целевого репозитория. Канонический список (одна точка истины —
задокументирован здесь и повторён байт-в-байт в `launchers/cc-config.cmd`/`cc-config.sh` и в
подсказках `cc-doctor`):
- `Bash(pwsh -File tools/codex-runtime.ps1 *)` — фактическая команда адаптеров (её и
  проверяют пер-сессионный гейт Фазы 1.1 processor'а и `cc-doctor`);
- `Bash(codex exec *)` — исторический якорь.

Если `.claude/settings.local.json` ещё нет — `cc-config` создаёт его с этим списком. Если
файл уже существует, но какого-то из канонических правил в нём нет, `cc-config` **домерживает**
только недостающее правило (или оба) в существующий массив `permissions.allow` (add-only: остальные ключи,
существующие allow-правила, deny-списки, hooks сохраняются; запись атомарная — temp-файл +
подмена), задача T-078. Это осознанный пересмотр T-058 (тот убрал прежний авто-домердж,
опасаясь «молча раздуть» файл разрешений оператора): домердж восстановлен **узко** — только
добавление недостающего правила и только при явном ручном запуске launcher'а оператором
(тот же consent, которым уже оправдано создание файла с нуля); launcher никогда не
переписывает и не удаляет чужое содержимое. Идемпотентность: если правило (по подстрочной
pre-проверке) уже присутствует — файл не меняется; повторный запуск дублей не создаёт.
Вырожденные случаи (файл — невалидный JSON, `permissions`/`allow` не той формы, файл
недоступен для записи, `jq` недоступен в `.sh`-варианте) не приводят к тихому отказу:
launcher печатает точную инструкцию добавить правило вручную в `permissions.allow`.
`cc-doctor`
проверяет то же разрешение read-only: если включён хотя бы один из эффективных
`CODEX_CODER`/`CODEX_REVIEWER` (с учётом фолбэка на переменную окружения) **или**
`CODEX_CIFIX` (только из `config.md`) — т.е. активна любая из трёх Codex-маршрутизаций — а
разрешение не подтверждено (ни сессионным признаком `CC_CODEX_EXEC_GRANT`, ни allow-правилом
в массиве `permissions.allow` файлов `.claude/settings.local.json`/`.claude/settings.json`
проекта; совпадения в `deny`/`hooks`/комментариях не считаются) — выдаёт `WARN` с точной
подсказкой; при всех трёх выключенных ключах разрешение не требуется и `cc-doctor` печатает
`OK`; сам `cc-doctor` ничего не создаёт и не правит. То же разрешение (тем же единым
контрактом: сессионный грант launcher'ов `CC_CODEX_EXEC_GRANT`, иначе строгий разбор
`permissions.allow`) проверяет и разовый на сессию permission-гейт Фазы 1.1 processor'а. Если конкретная версия Claude Code всё же
отклонит эту форму несмотря на allow-правило — запасной путь: PreToolUse-хук, возвращающий
`allow` для валидированной формы (`codex exec` c `--sandbox read-only|workspace-write`, без
`danger-full-access`). В текущей конфигурации хук **не задействован**: основной механизм —
сессионный грант launcher'ов плюс постоянное правило `cc-config`.

**Режим одобрения и fail-closed при отказе песочницы.** CLI-**флаг** `--ask-for-approval`
актуальная `codex exec` (`codex-cli` 0.142.5) не принимает — вызов с ним падает на разборе
аргументов (`error: unexpected argument '--ask-for-approval' found`, exit 2). Но
неинтерактивность `codex exec` сама по себе **не** делает исполнение fail-closed: под
дефолтной `approval_policy` (`on-request`/`on-failure`) отказ инициализации песочницы (на
Windows — `CreateProcessAsUserW failed: 5`, доступ запрещён на запуске restricted-token
процесса) молча эскалирует команду к запуску **без** изоляции. Чтобы закрыть этот fail-open
путь, оба адаптера (`coder_codex`/`reviewer_codex`) на **каждом** вызове пинят политику
**config-оверрайдом** `-c approval_policy=never` (не CLI-флагом — оверрайд `codex exec`
принимает). Это Orchestra-фиксируемая политика: она **не** зависит от значения
`approval_policy` в пользовательском `~/.codex/config.toml`. Под `never` codex не повышает
режим исполнения — команду, которую нельзя выполнить в песочнице (в т.ч. когда песочницу не
удалось поднять), он возвращает как сбой, а не запускает без изоляции. Адаптер распознаёт
сигнатуру отказа песочницы как ENV_LIMIT-класс `sandbox-init` и **до любых правок** штатно
эскалирует (`CODEX_FAILED — ENV_LIMIT/sandbox-init`), после чего processor откатывается на
Claude. Изоляция записи на Windows-restricted-token-песочнице **структурно не гарантируется** —
её держит именно эта проверяемая fail-closed связка (пиннинг политики + распознавание
`sandbox-init` + пост-проверка diff у `coder_codex`), а не структурные свойства песочницы.
Старые версии codex, где `exec` ещё принимал `--ask-for-approval` флагом, текущей формой не
поддерживаются (эквивалент задаётся config-оверрайдом).

**Механический протокол — в runtime (T-075).** Сборку argv, запуск `codex exec`, приём
промпта через stdin, раздельный захват stdout/stderr/RC, fail-closed `approval_policy=never`,
классификацию отказов (`sandbox-init`/`CreateProcessAsUserW failed: 5` и др.), порог
негабаритного diff, проверку reviewer-вывода, валидацию брокер-команд, гарантию отсутствия
коммитов и очистку только своей рабочей копии выполняет единый исполняемый скрипт
`tools/codex-runtime.ps1` (кросс-платформенный, по образцу `tools/queue-tx.ps1`), который оба
адаптера зовут коротким контрактом. Ролевые файлы `coder_codex`/`reviewer_codex` оставляют за
собой только выбор контекста и оценку содержательных результатов; поведение (сентинелы,
нормализованный вызов, отсутствие коммитов) сохранено. Тесты — `tests/test-codex-runtime.ps1`.

**Что можно делегировать:**
- **Реализация и `R-`-фиксы** (`coder_codex`, ключ `CODEX_CODER`) — задачи низкой/средней
  сложности.
- **Ревью** (`reviewer_codex`, ключ `CODEX_REVIEWER`) — независимое ревью. Даёт не только
  экономию, но и **разнообразие моделей** (другая модель ловит иные дефекты).
- **Точечные CI/сборочные фиксы** (`coder_codex` Режим 3, ключ `CODEX_CIFIX`) — обычно
  механические (линт/типы/мелкий тест).

**Независимость (важно).** `reviewer_codex` применяется **только к задачам,
реализованным Claude**; если задачу реализовал codex — ревью остаётся на Claude. Так
«другой ум» гарантированно касается каждой задачи. Практическое следствие: при обоих
флагах (`CODEX_CODER`+`CODEX_REVIEWER`) на одном уровне задача получает ровно один шаг на
codex и один на Claude.

**Рекомендуемая комбинация для максимальной экономии Opus:** `CODEX_CODER: off` (или
`fast`) + `CODEX_REVIEWER: fast+std` — задачи уровня `coder` идут «Claude-реализация +
Codex-ревью», снимая самый дорогой повторяющийся расход (opus `reviewer`), а
Opus-квота остаётся под `coder_deep`.

- `CODEX_CODER` — какие уровни реализации идут в codex: `off` | `fast` (только
  `coder_fast`) | `fast+std` (`coder_fast` и `coder`). `coder_deep` — **никогда**.
  Фолбэк на переменную окружения `CODEX_CODER` (см. ниже).
- `CODEX_REVIEWER` — какие уровни ревью идут в codex (только для Claude-реализованных
  задач): `off` | `fast` (`coder_fast`) | `fast+std` (`coder_fast` и `coder`) | `deep`
  (как `fast+std`, плюс для `coder_deep` — дешёвый диверсити-проход **в дополнение** к
  Opus-ревью, не вместо него). Фолбэк на переменную окружения `CODEX_REVIEWER` (см. ниже).

**Фолбэк на переменные окружения.** Только `CODEX_CODER` и `CODEX_REVIEWER` (остальные
ключи — исключительно из `config.md`) разрешаются по цепочке `config.md` → переменная
окружения ОС → дефолт `off`. Значение из окружения проходит ту же валидизацию, что и из
файла; пустое/нераспознанное — как незаданное. Выставить окружение:

```sh
# POSIX / bash (в профиле или перед запуском)
export CODEX_CODER=fast
export CODEX_REVIEWER=fast+std
```
```powershell
# Windows PowerShell (текущая сессия)
$env:CODEX_CODER = 'fast'
$env:CODEX_REVIEWER = 'fast+std'
# постоянно для пользователя:
[Environment]::SetEnvironmentVariable('CODEX_REVIEWER', 'fast+std', 'User')
```
- `CODEX_CIFIX` — `on` включает точечные CI/сборочные фиксы (Режим 3) через codex
  (Фазы 4.3 и 5.4). Интеграционные `F-` (Фаза 5.2) codex не ведёт никогда.
- `CODEX_MODEL` — модель codex (`-m`). Пусто → модель из `~/.codex/config.toml`
  (устойчиво к смене имён моделей). Пример: `gpt-5-codex` — уточните точную строку для
  вашей версии codex (`codex --model` / страница моделей).
- `CODEX_REASONING` — усилие рассуждения (`model_reasoning_effort`): `auto` (реализация:
  `coder_fast→low`, `coder→medium`; ревью: `→medium`) | `low` | `medium` | `high`.
- `CODEX_SANDBOX` — песочница codex для `coder_codex` (`--sandbox`). `workspace-write`
  (рекоменд.): правки в рабочем дереве; исходящая сеть — по ключу `CODEX_NETWORK` (ниже).
  `reviewer_codex` всегда работает в `read-only` (ревью не мутирует код). На Windows codex
  исполняется в ослабленной restricted-token-песочнице; отдельной admin-настраиваемой
  `elevated`-песочницы больше нет — фича `elevated_windows_sandbox` из codex удалена. Эта
  песочница на Windows **структурной** изоляции записи не гарантирует, поэтому границу держит
  fail-closed политика `-c approval_policy=never` на каждом вызове (см. «Режим одобрения и
  fail-closed при отказе песочницы» выше): при сбое инициализации песочницы команда не
  запускается вне неё, а адаптер эскалирует `ENV_LIMIT/sandbox-init` и работа откатывается на
  Claude.
- `CODEX_NETWORK` — исходящий сетевой доступ песочницы `coder_codex`: `on` (по умолч.) |
  `off`. Ключ читает **только** `coder_codex` (реализация/`R-`/Режим 3); `reviewer_codex`
  его игнорирует (ревью read-only, сеть не нужна).
  - `on` (дефолт) — `coder_codex` добавляет к вызову `codex exec` оверрайд
    `-c sandbox_workspace_write.network_access=true` (проверено на codex-cli 0.142.5: без
    него исходящие соединения блокируются) и пробрасывает git на openssl-бэкенд через
    `-c shell_environment_policy.set={GIT_CONFIG_COUNT="1",GIT_CONFIG_KEY_0="http.sslBackend",GIT_CONFIG_VALUE_0="openssl"}`;
    constraints-блок промпта при этом описывает доступную сеть. Серия целенаправленно
    делает сетевые задачи (обновление зависимостей, регенерация lock-файлов, скачивание
    ресурсов) штатными для codex, а не отдаёт их Claude.
  - `off` — вызов и constraints-блок промпта не меняются: полностью офлайн-песочница
    (строгая изоляция для тех, кому она нужна).
  - **Проверенная TLS-матрица Windows** (песочница = restricted token; schannel в ней не
    работает — `AcquireCredentialsHandle failed: SEC_E_NO_CREDENTIALS`; опция
    `sandbox_permissions=["disk-full-read-access"]` НЕ помогает — ограничение на уровне
    LSA, не файлов):
    - node/npm (свой OpenSSL) — работает напрямую (проверено, HTTPS 200);
    - python/pip, uv (OpenSSL/rustls) — тот же класс, ожидаемо работают;
    - git — работает **только** с `http.sslBackend=openssl` (адаптер пробрасывает его
      автоматически при `on`); дефолтный schannel-бэкенд падает;
    - cargo (libcurl+schannel, без переключения бэкенда) — в песочнице НЕ работает даже с
      сетью; его сетевые шаги идут через брокер (T-063), не в песочнице.
  - **Риск.** `on` — дефолт, ослабляющий изоляцию: песочница `coder_codex` получает
    исходящий сетевой доступ, и codex во время реализации задачи может обращаться к
    произвольным внешним хостам. Кому нужна строгая офлайн-изоляция — ставит `off`.
- `CODEX_CMD` — бинарь/команда codex (если не в PATH или через обёртку). Передаётся
  runtime-обёртке аргументом `--codex-cmd`, поэтому под предвыданный грант launcher'ов
  (`Bash(pwsh -File tools/codex-runtime.ps1:*)`) попадает и нестандартное значение —
  **собственного** allow-правила оно не требует (см. «Разрешение на запуск codex» выше).

**Сетевой брокер зависимостей (`coder_codex`).** Даже при `CODEX_NETWORK: on` TLS части
инструментов в песочнице Windows мёртв (cargo/schannel, дефолтный schannel-git — см. матрицу
выше), поэтому обновить их зависимости внутри codex нельзя. Брокер закрывает эту брешь:
адаптер `coder_codex` **сам** (не codex) выполняет строго ограниченный **allowlist**
канонических lock/fetch-команд по экосистемам — `cargo update`/`cargo fetch`,
`npm install`/`npm ci`, `pip install -r`/`pip download`, `uv lock`/`uv sync` (расширяемо) — в
worktree задачи **вне** песочницы, где TLS и сеть рабочие. Это регенерирует lock-файлы
(легитимная часть результата задачи) и наполняет кэш (`~/.cargo` и т.п.), который песочница
codex затем читает для офлайн-сборки. Никаких произвольных сетевых команд: всё вне allowlist
отклоняется. Триггеры — **(а)** детерминированный (codex изменил манифест зависимостей, а
сборка/smoke упали с сигнатурой `network`/`tls-schannel` из ENV_LIMIT-классификации) и
**(б)** явный (codex завершается строкой `NEED_NET: <команда>`, адаптер валидирует её по
allowlist). Брокер-шаг **не** тратит бюджет адаптерных итераций codex; на задачу — не более
2 брокер-циклов, дальше эскалация тем же классом `CODEX_FAILED — ENV_LIMIT/<класс>`. Брокер
VCS **не мутирует** (не коммитит, не пушит) — гарантия «без коммитов» соблюдается. **Связь с
`CODEX_NETWORK` (T-061):** `on` даёт песочнице сеть для OpenSSL-инструментов
(npm/pip/uv/openssl-git прямо в песочнице), а брокер добирает остаток (schannel/cargo); при
`off` брокер — единственный путь к сети для любых зависимостей. Детали механизма — в
`agents/coder_codex.md`. `reviewer_codex` брокера **не имеет** (ревью read-only, сетевых шагов
нет).

**Маршрутизация сетевых задач (T-064).** planner проставляет дескриптору задачи с сетевым
признаком поле `Сеть: требуется` + `Экосистема: <cargo|npm|pip|uv|прочее>` (см.
`agents/planner.md`, «Признак сети»); processor перед тем как отдать задачу `coder_codex`,
сверяет по этому полю, реально ли доступен сетевой путь для экосистемы — класс сетевой
задачи → путь codex:

| Класс (`Экосистема:`) | Путь сети у `coder_codex` | `coder_codex` доступен, если |
|---|---|---|
| `cargo`/`npm`/`pip`/`uv` (манифест/lock менеджера зависимостей) | сетевой брокер — обслуживает эти экосистемы **безусловно**; плюс прямая сеть песочницы при `CODEX_NETWORK: on` для npm/pip/uv/openssl-git | всегда — брокер не зависит от `CODEX_NETWORK` |
| `прочее` (сетевые тесты, скачивание внешних ресурсов, произвольные сетевые вызовы — вне allowlist брокера) | только прямая сеть песочницы | `CODEX_NETWORK: on` |

При недоступном пути (класс `прочее` и `CODEX_NETWORK: off`) processor не запускает
`coder_codex` вовсе — сразу использует Claude-исполнителя уровня задачи, без пустого
codex-прогона. Отсутствие поля `Сеть:` в дескрипторе — прежнее поведение (маршрутизация
только по `CODEX_CODER`). Детали резолвера — в `agents/processor.md`, «Codex-исполнитель
(coder_codex) и маршрутизация».

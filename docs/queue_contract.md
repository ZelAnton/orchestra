# Контракт постановки задач в очередь `.work/Tasks_Queue.md`

Это **единый нормативный источник** механических правил, по которым в очередь
`.work/Tasks_Queue.md` пишут задачи все её популяторы. Правила ниже одинаковы для
всех и живут только здесь: сами файлы популяторов их не переизлагают, а ссылаются
на этот документ и несут лишь свой уникальный мандат (что именно и когда ставить в
очередь, свои источники анализа и специфичный дедуп).

Документ обязателен для пяти агентов-популяторов:

- `agents/queue_builder.md` — превращает пользовательский источник/формулировку в
  записи очереди;
- `agents/code_auditor.md` — заводит находки-дефекты исходного кода;
- `agents/enhancement_scout.md` — заводит идеи развития проекта;
- `agents/github_sync.md` — заводит задачи из issues/PR;
- `agents/thinker.md` — оформляет согласованные с пользователем выводы в задачи.

Очередь обрабатывает конвейер оркестратора `processor`: он захватывает батч
параллельно-безопасных задач, `planner` выбирает и планирует их, исполнители
реализуют каждую в своём git-worktree, ревьюеры проверяют, а `processor` коммитит,
сливает и публикует. От корректности записей напрямую зависит работа конвейера.

Полная каноническая спецификация формата (с расширенными примерами и чеклистом
перед сдачей очереди) — `~/.claude/specs/Tasks_Queue_Format.md`; этот документ —
её нормативная выжимка внутри репозитория, достаточная для повседневной постановки
задач. При расхождении между ними приоритет у настоящего файла.

## 1. Форма заголовка записи

Каждая задача — один блок. Первая строка (заголовок) строго по шаблону:

```
### [T-NNN] Краткое название — статус: не начата
<тело: связное описание сути задачи>
```

- Начинается ровно с `### ` (уровень H3); уровень заголовка не менять.
- Содержит `[T-NNN]` — идентификатор в квадратных скобках (правила нумерации — §2).
- Краткое название — одна строка в **императиве** («Добавить…», «Исправить…»,
  «Вынести…»), по сути задачи.
- Разделитель ` — ` (тире с пробелами) и точно `статус: не начата`.
- Между соседними записями — одна пустая строка.

## 2. Идентификатор `T-NNN`

- `NNN` — сквозной порядковый номер; рекомендуются ведущие нули (`T-001`, `T-014`).
- ID **уникален и никогда не переиспользуется**. Выполненные задачи удаляются из
  очереди, поэтому по одному текущему файлу следующий номер не вычислить: возьми
  **максимальный** `T-NNN` среди **всех трёх** источников —
  `.work/Tasks_Queue.md`, `.work/Tasks_Done.md` и `.work/tasks/*/task.md` — и
  продолжи нумерацию от него. Не ориентируйся только на записи, оставшиеся в очереди.

## 3. Статусы

- Новые задачи создаёшь **только** со статусом `не начата`, без суффиксов.
- Прочие формы заголовка проставляет **только** конвейер (`processor` через
  `executor`); ты их не создаёшь и записи с ними не трогаешь (см. §5). Раздел
  приведён, чтобы ты умел их **узнавать**:
  - **Захват в работу**: `— статус: в работе · батч=B-… ·
    worktree=.work/worktrees/<T-ID> · ветка=task/<T-ID>`.
  - **Возврат в очередь после карантина**: `— статус: не начата · попытка=N ·
    карантин=<причина>`. **Базовый статус здесь по-прежнему `не начата`** — суффиксы
    `· попытка=N` и `· карантин=…` его не меняют.
  - **Терминальная эскалация**: `— статус: эскалирована · причина=…`. Терминальна:
    из очереди не удаляется, но и в работу не берётся.
- **Статус определяй по вхождению подстроки `статус: не начата`.** Запись
  `не начата · попытка=N · карантин=…` — валидный кандидат на исполнение; не путай
  её с чужим/терминальным статусом. **Не срезай, не переставляй и не «нормализуй»**
  суффикс `· попытка=N · карантин=…`: `попытка=N` — счётчик карантинных попыток, по
  которому оркестратор решает, эскалировать ли задачу; его потеря обнулит счётчик и
  зациклит карантин.

## 4. Тело задачи

Тело самодостаточно: `planner` дословно копирует его в дескриптор
`.work/tasks/<T-ID>/task.md`, по которому работает исполнитель и сверяется ревьюер.
Оно должно быть понятно без доступа к тебе и внешнему контексту.

- **Суть и цель** — *что* нужно сделать и *зачем* (контекст), границы задачи и
  известные ограничения — связной прозой.
- **Конкретные файлы/модули** указывай, когда это выводимо из источника, — `planner`
  опирается на эти пути, чтобы не открывать собственную разведку репозитория. Если
  источник действительно слишком расплывчат, чтобы вывести пути, — не выдумывай их,
  а явно пометь в теле «источник слишком расплывчат для локализации».
- **Оценки** (сложность/effort/ответственность, уровень исполнителя, конфликт-домен)
  и формальные критерии выполнения авторитетно задаёт `planner` при захвате задачи —
  их не пиши.
- **Разбивку на этапы (`## План выполнения`) по умолчанию оставляй `planner`'у** —
  большинство популяторов описывают только суть, без плана. Исключение: популятор,
  чей мандат это явно допускает (см. его файл — например, `thinker`, который
  исследует код перед постановкой), может для крупной задачи сразу дать
  `## План выполнения` этапами-чекбоксами `- [ ]` (все невыполненные). Если план не
  дан, `planner` составит его сам.
- **Критерии готовности в теле — опциональны.** Формальные критерии выполнения
  авторитетно формирует `planner` (`## Критерии выполнения`), поэтому требовать их
  в каждой записи не нужно. При желании можешь дать черновой ориентир прямо в прозе
  описания, но **не** отдельным заголовком `## Критерии готовности` /
  `## Критерии выполнения` — иначе он задвоится с разделом, который `planner` добавит
  сам поверх дословно скопированного тела.

## 5. Размер, порядок и правила пополнения

- **Одна задача = один целостный, самостоятельно ревьюируемый и коммитируемый
  результат.** Крупное дроби на **несколько отдельных записей**, а не в одну огромную
  задачу; мелкие однотипные правки логично объединять.
- **Порядок записей = порядок выполнения**: задачи-предпосылки ставь **выше**
  зависящих от них. Обычно дописывай новые задачи в конец в разумном порядке; если
  новая задача — предпосылка уже стоящей, помести её перед зависимой.
- При пополнении **не изменяй и не переставляй** уже имеющиеся записи и **не трогай**
  ни одной задачи со статусом, отличным от `не начата` (сверяй по вхождению
  `статус: не начата`, см. §3 про карантинный суффикс). Только добавляй новые.
- Если файла `.work/Tasks_Queue.md` нет — создай его (заголовок `# Очередь задач`,
  затем записи).

## 6. Дедуп: три источника

Перед записью убедись, что такой задачи **ещё нет** и она **не выполнена** — не
дублируй. Сверяйся со **всеми тремя** источниками:

1. `.work/Tasks_Queue.md` — уже стоящие в очереди задачи;
2. `.work/Tasks_Done.md` — уже выполненные задачи;
3. `.work/tasks/*/task.md` — **захваченные, но ещё не завершённые** задачи (в работе).

Третий источник закрывает пробел: задача по уже захваченной, но не выполненной работе
не должна предлагаться повторно (в очереди её заголовок уже несёт статус захвата, а
не `не начата`, но в дескрипторе `.work/tasks/<T-ID>/task.md` она видна как задача в
работе — сверяйся именно по сути, а не только по строке очереди).

Если у популятора есть **свой дополнительный** источник дедупа (например, у
`github_sync` — файл `.work/Github_Sync.md` с маппингом issue/PR ↔ T-ID), он
применяется как **дополнение** к общему дедупу по трём источникам, а не как его
замена. При дрейфе или устаревании такого специфичного источника общий дедуп по трём
источникам остаётся рабочим предохранителем.

## 7. Поведение под `.work/orchestrator.lock`: inbox

Перед записью в очередь проверь `.work/orchestrator.lock`. Если он существует —
`processor` активен и владеет очередью (`.work/Tasks_Queue.md`/`.work/Tasks_Done.md`):
прямая запись через read-modify-write рискует затереть его правки (метку захвата,
удаление выполненной записи) или переиспользовать только что освободившийся `T-NNN`.

**Не пиши очередь напрямую при активном локе — вместо этого положи предложение в
inbox.** Транзакционный интерфейс (§9) для каждой операции постановки предлагает две
формы:

- **лока нет** → `queue-tx propose …` пишет запись в очередь сразу (под собственным
  коротким локом транзакции, безопасно относительно других популяторов);
- **лок есть** → `queue-tx inbox-add …` складывает предложение отдельным файлом в
  `.work/queue_inbox/` и **очередь не трогает**. `processor` вычитывает inbox
  (`queue-tx inbox-drain`) на безопасной границе когорты и штатно ставит записи в
  очередь — без остановки обработки текущей когорты и **без потери** твоего
  предложения. Дедуп (§6) и валидация зависимостей (§11) применяются при вливании так
  же, как при прямой постановке.

Так активная когорта больше не блокирует приём предложений. Если интерфейс `queue-tx`
в среде недоступен (нет `pwsh`/`powershell`), деградируй к прежнему поведению: **не
пиши очередь**, а покажи готовые блоки задач в чате и предложи добавить их, когда
`processor` освободится. Не пиши «в стол» и не теряй результат разбора.

## 8. Чего не делать

- Не проставляй статусы, кроме `не начата`, для новых задач; не переиспользуй и не
  меняй ранее выданные `T-NNN`.
- Не редактируй `.work/Tasks_Done.md`, `.work/tasks/*/task.md`, журналы ревью
  (`.work/tasks/*/review.md`, `.work/review_integration.md`) и исходный код —
  репозиторий ты только читаешь (кроме файлов, явно закреплённых за конкретным
  популятором, — например, `.work/Github_Sync.md` у `github_sync`).
- Не добавляй в очередь чекбоксы статуса/прогресса, `SUMMARY`-записи и что-либо из
  журнала ревью — очередь это только бэклог.
- Не пиши в задачах про ИИ/агента/Claude, не вставляй тексты коммитов и VCS-команды —
  это зона исполнителя.

## 9. Транзакционный интерфейс очереди (`tools/queue-tx.ps1`)

Все мутации очереди идут через **единый транзакционный интерфейс** —
`tools/queue-tx.ps1` (вызывается `pwsh -File tools/queue-tx.ps1 <операция> --work
<абс .work> …`; работает и под Windows PowerShell 5.1). Он — источник правды
механики: каждую операцию он выполняет под собственным коротким локом
(`.work/queue-tx.lock`, атомарный lock-файл, отдельный от `.work/orchestrator.lock`),
пересчитывая ID и дедуп **по текущему состоянию на диске под этим локом**, и пишет
крэш-безопасно (temp-файл + атомарный rename). Поэтому **повторный `T-NNN`,
потерянное обновление (lost update) или повреждённый заголовок не могут возникнуть в
результате нормальной работы интерфейса** — даже при нескольких параллельных
писателях. Ручной read-modify-write очереди в обход интерфейса не используй.

Операции (у каждой — чёткие предусловия/эффекты; полное описание и коды выхода — в
шапке `tools/queue-tx.ps1`):

- `allocate-id` — вернуть следующий свободный `T-NNN` (максимум по трём источникам §2, +1).
- `propose --title <…> [--body/--body-file <…>] [--predecessors "T-a,T-b"]
  [--expected-generation N] [--id T-NNN]` — идемпотентный дедуп (§6) + валидация графа
  зависимостей (§11) + постановка новой записи `не начата`. Отклоняет дубль (код 4),
  неверную зависимость (код 5), устаревшее поколение (код 3, см. §10).
- `inbox-add …` (те же аргументы, что у `propose`) — положить предложение в
  `.work/queue_inbox/` без записи очереди (для активной когорты, §7).
- `inbox-drain` — влить накопленные предложения inbox в очередь (вызывает
  `processor` на границе когорты); дедуп/валидация применяются к каждому.
- `capture --id T-NNN --batch B-… [--worktree …] [--branch …]` — `не начата` →
  `в работе · батч=… · worktree=… · ветка=…`, сохраняя суффикс `· попытка=N` (§3).
- `return --id T-NNN --reason <…> [--max-attempts N]` — возврат после карантина:
  `не начата · попытка=<N+1> · карантин=<…>`, а при исчерпании попыток — терминальная
  `эскалирована` (мэппинг Фазы 6.2 processor'а).
- `escalate --id T-NNN --reason <…>` — терминальная `эскалирована · причина=<…>`.
- `archive --id T-NNN` — удалить запись `[T-NNN]` из очереди целиком (идемпотентно).
- `validate-deps` — проверить весь граф зависимостей (§11); код 5 при находках.
- `ready [--id T-NNN]` — резолвер готовности (§12).
- `generation` / `bump-generation` — прочитать/инкрементировать поколение (§10).

**Кто чем пользуется.** Популяторы (§0) — `allocate-id`, `propose`/`inbox-add`;
`planner` — `ready`/`validate-deps` при выборе кандидатов; `processor` —
`capture`/`return`/`escalate`/`archive` и `inbox-drain` на границах когорты
(механические правки строк очереди processor по-прежнему может делегировать
`executor`'у по готовому тексту — но авторитетную транзакцию с поколением и
валидацией даёт интерфейс).

## 10. Поколение состояния (generation counter)

Интерфейс ведёт монотонный счётчик **поколения** очереди в `.work/queue_state.json`
(инкремент на каждую успешную мутацию). Он — аналог version/generation counter для
оптимистичной сверки (compare-and-swap):

- прочитал очередь на поколении `G` → подготовил предложение → вызвал
  `propose --expected-generation G`. Если к моменту коммита поколение уже не `G`
  (очередь изменилась под тобой), транзакция **отклоняется** (код 3), а не молча
  перезаписывает поверх чужого изменения; перечитай состояние и повтори.
- флаг `--expected-generation` **опционален**: даже без него интерфейс под своим локом
  пересчитывает ID/дедуп по актуальному состоянию и только **дописывает** запись
  (никогда не перезаписывает поверх), поэтому дубль/lost update исключены и без CAS;
  CAS даёт раннее, явное обнаружение «мир изменился» для повторной сверки.

## 11. Декларативные зависимости: поле `Предпосылки:`

Задача может декларативно указать обязательные **предпосылки** (predecessor) —
машиночитаемым полем в **теле** записи (формат заголовка §1 не меняется, поэтому
писатели/читатели, парсящие только заголовок, остаются совместимы):

```
### [T-050] Переключить контроллеры на новый PaymentService — статус: не начата
<описание сути…>
Предпосылки: T-048, T-049
```

- Поле — отдельная строка `Предпосылки: T-a, T-b` (принимается и англ. алиас
  `Predecessors:`); перечисляет `T-ID`, которые должны быть **успешно завершены**
  раньше. Ставь предпосылки **выше** зависимой и в самой записи (порядок в файле —
  по-прежнему подсказка, но граф теперь явный, а не «на глаз»).
- **Валидатор** (`propose`/`validate-deps`) не даёт попасть в очередь записи с
  битым графом и называет причину: отсутствующий/несуществующий `T-ID`
  (`missing`), ссылка на себя (`self-reference`), цикл в графе (`cycle`), заведомо
  невыполнимая зависимость — предпосылка уже `эскалирована` и никогда не завершится
  (`infeasible`). Предпосылка, лежащая в `Tasks_Done.md`, — валидна (уже выполнена).

## 12. Резолвер готовности (readiness)

`queue-tx ready` (им пользуется `planner` при выборе кандидатов) допускает задачу к
захвату **только когда все её предпосылки успешно завершены** — критерий «завершена» =
запись предпосылки лежит в `.work/Tasks_Done.md` (задача опубликована и заархивирована).
Пока хоть одна предпосылка не завершена, задача **не готова**, и резолвер называет
**конкретную** блокирующую предпосылку (`waiting on T-048` / `blocked: T-049 escalated`
/ `missing predecessor T-050`), а не выводит это из текста задачи вручную. Независимые
задачи (без общих предпосылок) готовы **параллельно** и не блокируют друг друга.

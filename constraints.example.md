# Политика ограничений проекта — `.work/constraints.md`

Это **человекочитаемая политика ограничений** конкретного проекта: единый файл, который
роли конвейера читают, чтобы ограничить радиус изменений (blast radius) до того, как
что-то будет сделано или опубликовано. `launchers/cc-config` сеет его из этого шаблона
(`constraints.example.md`) в `.work/constraints.md` при первом запуске, **никогда не
перезаписывая** уже существующий; дальше файл редактирует человек под свой проект.

В отличие от `.work/config.md` (числовые ключи `KEY: value`, куда копируется только
выделенный блок-seed) здесь в `.work/constraints.md` копируется **весь документ целиком**
— он и есть содержимое политики. Правьте значения прямо в разделах ниже.

## Как это читают роли

- **processor** — **перед захватом новой когорты** сверяет конфликт-домены выбранных
  задач с запрещёнными путями; **повторно перед публикацией** сверяет фактически
  затронутые файлы, учитывает push/merge policy и обязательные проверки. Нарушение →
  отказ захватить/опубликовать **конкретную** задачу с понятным объяснением, **без
  падения** всего конвейера.
- **planner** — при нарезке задач: не берёт (или помечает) задачи, чей конфликт-домен
  пересекается с запрещёнными путями, и учитывает пороги размера при разбиении на этапы и
  оценке сложности.
- **coder** — избегает запрещённых путей при реализации; если задача действительно
  требует тронуть запрещённый путь — останавливается и эскалирует processor, а не правит
  молча. Push/merge не выполняет никогда (это делает processor по политике ниже).
- **reviewer** — сверяет затронутые diff-ом пути с запрещёнными как часть проверки границ
  diff; diff, залезающий в запрещённый путь, — это находка.

**Деградация без ошибок.** Если файла `.work/constraints.md` нет — конвейер ведёт себя как
раньше, никаких новых жёстких отказов. То же правило действует **пораздельно**: пустой
раздел (нет активных записей под заголовком, только пояснение и пример) = ограничение не
задано и не применяется. Значения по умолчанию ниже намеренно **минимальны и безопасны
для незнакомого проекта** (пустой denylist, а не запрет всего) — включайте ограничения
осознанно, по мере понимания проекта.

**Исполняемая граница (T-084).** Эта политика — не только текст «на прочтение»: её
применяют механически через CLI `tools/policy.ps1` (единый versioned schema source —
`tools/policy-schema.ps1`). processor и merger не доверяют текстовому отчёту там, где есть
проверка: `check-paths` сверяет фактически изменённые пути с denylist ниже (после каждого
возврата исполнителя и перед commit/merge/publication; запрещённая правка так не попадает в
интеграционный результат), `check-publish` применяет разрешённые ветки/remotes и push/merge
policy как технический precondition, а `guard-path` канонизирует путь и исключает выход за
корень через `..`/symlink/junction и подмену worktree. Разделы, оставленные пустыми, при
этом по-прежнему ничего не ограничивают (деградация без ошибок).

Полноценную risk-классификацию и парковку задач до решения человека (human gates) этот
файл пока **не** реализует — это отдельный этап развития (см. `plans/LOOP_ORCHESTRA_ROADMAP.md`).
Сейчас роли **читают** политику и применяют базовые ограничения (denylist, push/merge
policy, обязательные проверки, пороги, пометка human-review категорий) исполняемо, как
описано выше.

## Соглашение о формате

Каждый раздел ниже — одно ограничение. Активные значения — это **некомментированные
пункты списка** под подзаголовком «Активные ограничения» (или явная строка политики).
Блоки под подзаголовком «Пример» — только для ориентира и ролями как значения **не**
читаются. Пустой список активных значений (или «(пусто)») = ограничение не действует.

## Запрещённые пути (denylist)

Глоб-шаблоны путей, которые автоматический конвейер **не должен** менять и публиковать без

т.п.). По умолчанию **пусто** — ничего не запрещено (безопасный старт для незнакомого
проекта; заполняйте по мере понимания, что в проекте критично).

## Push/merge policy

Как конвейеру разрешено публиковать результат. По умолчанию — **как в `.work/config.md`**
(ключ `PUSH`): этот файл ничего не ужесточает. Установите явную политику, чтобы, например,

этапом; сейчас processor лишь учитывает заданную здесь политику и при запрете
останавливается перед публикацией с объяснением, не пуша.)

## Обязательные CI-проверки публикации

Имена **обязательных CI-проверок** (check/context в форже — GitHub Actions job, required
status check и т.п.), которые processor должен увидеть **зелёными на точном опубликованном
SHA**, прежде чем считать батч опубликованным и заархивировать задачи (Фаза 5.4). Это
**имена набора**, а не команды: processor привязывается к SHA пуша, собирает **весь**
перечисленный набор и **fail-closed** трактует отсутствующий/незавершённый/отменённый/
красный результат как «не готово к публикации» (тайминг ожидания — `PUBLISH_CI_DEADLINE_SEC`/
`PUBLISH_CI_BACKOFF_SEC` в `.work/config.md`). По умолчанию **пусто** — обязательного набора
нет, ожидание CI управляется только `CI_WATCH` (прежнее поведение; деградация без ошибок).

**Активные ограничения** (по умолчанию пусто):

- (пусто — обязательных CI-проверок публикации нет; ожидание CI по `CI_WATCH`)

## Категории обязательного human review

Категории изменений, которые всегда требуют человеческого суждения (их нельзя свести к
механической проверке). planner и reviewer **помечают** задачу, попавшую в такую
категорию; полноценные human gates (парковка задачи до решения человека) вводятся
отдельным этапом — пока это сигнал для повышения внимания/уровня исполнителя и явной
пометки в отчёте, а не жёсткий отказ. Ниже — консервативный рекомендованный набор; уточните
под проект.

**Активные категории** (рекомендованный набор по умолчанию):

- изменение архитектурных границ и публичных контрактов/API;
- безопасность, auth, permissions, секреты, PII;
- платежи, production-инфраструктура, миграции данных;
- major-обновления зависимостей и широкий diff lock-файлов;
- изменение самой политики (`.work/constraints.md`) или обязательных проверок.

# Намерение: детерминированный компилируемый оркестратор с TUI

> Статус: **предложение / намерение**, не действующий runtime-контракт. Документ
> фиксирует замысел и обоснование перехода оркестратора Orchestra из семантического
> агента (`agents/processor.md`) в детерминированный компилируемый движок с
> операторским TUI. Он не отменяет `LOOP_ORCHESTRA_ROADMAP.md` и
> `OBSERVABILITY_PLATFORM_PLAN.md`, а уточняет и приближает их пункт о выносе
> детерминированных частей в код (roadmap §7.6).

## 1. Намерение одной фразой

Заменить LLM-дирижёра `processor.md` **детерминированным компилируемым движком**
(мозг) плюс тонким **TUI** (пульт), сохранив листовых агентов (planner, coder,
reviewer, merger, full_reviewer, populators, knowledge_curator) как LLM и сохранив
существующий транзакционный слой инструментов (`tools/*.ps1`) как есть.

Детерминированным становится **оркестратор** (управляющий поток), а не **система**:
листовые шаги остаются недетерминированными вызовами моделей. «Детерминированный
оркестратор» означает, что при данном наборе структурированных выходов листовых
агентов управляющий поток воспроизводим и тестируем, — а не то, что одна очередь
дважды даёт одинаковый код.

## 2. Контекст и ключевая находка

Orchestra уже сделала тяжёлую половину этой работы. Все критические переходы
состояния вынесены из промпта в тестируемые, крэш-восстановимые транзакционные
PowerShell-инструменты. Дизайн-философия зафиксирована прямо в шапке
`tools/harness.ps1`: «the control loop is an LLM prompt, but every one of its critical
state transitions is mediated by an executable transactional tool».

Уже детерминировано и покрыто тестами:

| Инструмент | Роль |
|---|---|
| `tools/queue-tx.ps1` | мутации очереди (атомарный лок, generation/CAS, inbox) |
| `tools/state-tx.ps1` | аренда владельца + гард переходов статуса + CAS поколения |
| `tools/outbox.ps1` | append-only `events.jsonl` |
| `tools/supervisor.ps1` | ограничение вызова исполнителя по дедлайну/попыткам/объёму/бюджету; 4-классовая остановка; resume-checkpoint |
| `tools/policy.ps1` + `tools/policy-schema.ps1` | валидация config/constraints, path-guards, publish-gates |
| `tools/codex-runtime.ps1` | безопасная сборка argv, маппинг сентинелов, порог diff |
| `tools/harness.ps1` | герметичный end-to-end прогон когорты по реальным инструментам с fault-injection и timestamp/event-id-независимым fingerprint конечного состояния |

Следствие: `agents/processor.md` (243 КБ, фазы 0–6) сегодня — это **дирижёр**,
который секвенсирует эти инструменты и принимает решения между ними. Значительная
часть guard-логики (`state-tx check-transition`, CAS поколения) существует по сути
как защита *от самого оркестратора*, потому что оркестратор — LLM, способный
недочитать, пропустить шаг или «сымпровизировать». Заменив LLM-дирижёра кодом, мы
устраняем именно тот класс отказов, ради которого построен защитный слой.

## 3. Решение

1. **Начать проект** детерминированного оркестратора — но не с TUI и не с big-bang
   переписывания `processor.md`, а с де-рискующего спайка и ядра-движка (см. §8–§9).
2. **Оставить листовых агентов LLM** — они пишут код и выносят инженерные суждения;
   это ирредуцибельно.
3. **Сохранить `tools/*.ps1`** как транзакционный слой — вызывать подпроцессами;
   нативно портировать только при доказанной необходимости.
4. **`processor.md` — оракул на время миграции**, cutover по fingerprint
   `harness.ps1`, после — вывод из эксплуатации.

Оставлять `processor` семантическим агентом *навсегда* — более слабый вариант:
243 КБ прозы с ежедневным churn уже стоят на пороге хрупкости, за которым сам
roadmap (§7.6) предписывает выносить детерминированные части в код.

## 4. Что такое «оркестратор» как компилируемая вещь

Главный принцип: **оркестратор — не модель и не «поумнел».** Он не пишет код, не
судит diff, не оценивает качество. Это долгоживущий детерминированный
процесс-дирижёр. Всё «мышление», которое сегодня делает `processor.md`, при
компиляции распадается ровно на две категории:

- **(а) чистые функции / деревья решений** над структурированными данными — таких
  большинство: выбор тира ревьюера по полям дескриптора, Codex-маршрутизация,
  «чисто/с находками» по наличию свежей `SUMMARY-R` и открытых `R-`, переизбрание
  ревьюера по полю `Реализовано:`, лимиты циклов по счётчикам, планирование когорты
  по непересекающимся conflict-доменам, бюджет/circuit-breaker.
- **(б) вызов наружу к LLM-листовому агенту** как к супервизируемому субпроцессу.
  Оркестратор их запускает и разбирает их структурированный ответ, но сам суждения
  не выносит.

Конкретный внутренний состав движка:

| Компонент | Что делает | Откуда берётся |
|---|---|---|
| Фазовый цикл 0–6 | явный state machine: аренда → открытие когорты → раунды исполнения → интеграция → джойн/публикация → учёт/очистка | портируется из фаз `processor.md` |
| Транзакционный слой | атомарные мутации очереди/аренды/статусов/outbox с CAS и крэш-восстановлением | уже готово (`tools/*.ps1`), вызов подпроцессами |
| Супервизор LLM-вызовов | спавн субагента, ограничение по дедлайну/попыткам/бюджету, классификация остановки, resume-checkpoint | `tools/supervisor.ps1` (см. риск R1) |
| Резолверы решений | тиринг ревью, Codex-маршрутизация, бюджет, планирование когорты | дерево решений из фаз → чистый код |
| Парсеры контрактов | разбор структурированных возвратов листовых агентов и `.work/`-артефактов | требует ужесточения контрактов (§8) |
| Store состояния | те же `.work/`-файлы + `events.jsonl` как источник истины для человека и TUI | уже есть |

То есть движок = **event-loop + state store + супервизор LLM-подпроцессов + деревья
решений**. Модель внутри него не живёт.

## 5. Два слоя

- **Оркестратор (движок)** — headless-процесс, который делает работу: держит аренду,
  гоняет когорты, спавнит листовых LLM-агентов, коммитит/мержит/публикует, пишет
  `.work/` и `events.jsonl`. Полностью функционален без UI (как сегодня `processor`
  функционален без `cc-status`).
- **TUI** — только представление и пульт поверх движка: читает
  `events.jsonl`/`status.md`, рисует обзор и Decision Inbox (экраны из
  `OBSERVABILITY_PLATFORM_PLAN.md` §6), обратно шлёт команды (pause, approve,
  force-lock, resume). Логики оркестрации не содержит; закрытие TUI не останавливает
  движок.

Разделение совпадает с уже спроектированной событийной архитектурой (events вверх,
commands вниз): TUI — локальный потребитель того же контракта, который в плане отдан
web/Android.

## 6. Технологический выбор

Гибрид по слоям, с однозначным выбором для каждого слоя.

| Слой | Чем писать | Почему |
|---|---|---|
| Движок оркестратора | **Rust** (первый выбор) или C#/.NET | надёжность, типобезопасность, долгоживущий процесс, конкуррентная супервизия N субагентов, property-тесты |
| TUI | **ratatui** (Rust) / **Spectre.Console** (C#) | первоклассные TUI-фреймворки в языке движка |
| Транзакционный слой | **PowerShell (оставить как есть)** | готов, оттестирован, крэш-восстановим, кросс-платформенный; нативный порт — только при узком месте |
| Драйвер листовых агентов | `claude -p --output-format stream-json` / `codex exec` как супервизируемые подпроцессы | язык-агностично; совпадает с моделью `tools/supervisor.ps1` |

### Почему не «скрипты» для ядра

Смысл затеи — уйти от хрупкости интерпретируемого, слабо структурированного
управляющего слоя. Ядро на pwsh/Python это подорвёт: долгоживущий stateful-демон,
конкуррентный спавн субагентов и отзывчивый TUI — не сильная сторона скриптов; у
PowerShell нет первоклассного TUI; типобезопасность и тестируемость state machine
слабее компилируемого языка. Граница чистая: **скрипты — для инструментов
(оставить), компилируемый язык — для движка и TUI (новое).**

### Почему не «оставить агентом»

Откладывание, а не решение. Единственная остающаяся роль агента — **оракул на время
миграции**.

### Rust vs C#/.NET

Оба легитимны и совпадают с линиями рабочего пространства (Rust-линия и .NET-линия).
Решает скорость команды, не теория. По умолчанию — **Rust**:

- один статический кросс-платформенный бинарь — чистая дистрибуция, аккуратно
  заменяет `cc-sync`-зеркалирование для движка;
- **ratatui** — вероятно лучший доступный TUI-фреймворк;
- культура корректности совпадает с одержимостью проекта крэш-восстановлением и
  инвариантами (типы, `Result`, `proptest`);
- совпадает с Rust-линией; в ней есть `agent-workspace` — **проверить первым**, по
  названию может оказаться смежным или фундаментом;
- `tokio` тянет долгоживущий демон + супервизию параллельных субагентов.

**Переключиться на C#/.NET, если** команда объективно быстрее пишет на .NET и готова
на более тяжёлый рантайм / single-file publish; Spectre.Console даёт хороший TUI.

### Agent SDK не должен диктовать язык

Соблазн взять TypeScript/Python ради официального Claude Agent SDK — не повод.
Листовых агентов гоним как супервизируемые подпроцессы через
`claude -p --output-format stream-json` и `codex exec` — язык-агностично и совпадает
с уже существующей моделью `tools/supervisor.ps1` (вызов = супервизируемый дочерний
процесс). Дверь к Rust/C# остаётся открытой; SDK стал бы обузой, тянущей движок в
скриптовый рантайм.

## 7. Что детерминировано, а что ирредуцибельно-LLM

Прочтение фазового алгоритма (например, пер-таск ревью 2.4–2.9 в `processor.md`)
показывает: «суждение» оркестратора — это в основном **дерево решений + разбор
структурированных маркеров**, а не творческое рассуждение:

- выбор тира ревьюера — функция от `Рекомендуемый исполнитель`, `REVIEWER_TIERING`;
- Codex-маршрутизация — дерево от config + поля `Сеть:` + KB-pitfalls + инварианта
  независимости maker/checker;
- «чисто/с находками» — по свежей `SUMMARY-R` и открытым `R-NN`;
- переизбрание ревьюера после фикса — по полю `Реализовано:`;
- лимиты — по полям `Циклов-ревью`/счётчикам.

Всё это компилируется в код. **Ирредуцибельно-LLM остаётся только внутри листовых
агентов** — они пишут код и выносят инженерные суждения.

Остаточный «мягкий» участок — там, где оркестратор сегодня интерпретирует свободный
текст отчёта листового агента. Детерминированный оркестратор требует, чтобы **каждый**
возврат листового агента был машинно-разбираемым без свободной неоднозначности (в
эту сторону уже идут `SUMMARY-R`/`R-NN`/`F-NN`/`events.jsonl`). Это реальная,
нетривиальная работа по ужесточению контрактов — и главный настоящий предпосыл.

## 8. Предпосылки (что сделать до/на старте)

1. **Спайк среды спавна субагентов** (единственная настоящая неизвестность). Написать
   минимальный процесс на целевом языке, который сам порождает и супервизирует один
   вызов `claude`-субагента (planner или coder) и один `codex exec` — вне Claude Code
   — с дедлайном, `maxTurns`, захватом структурированного вывода и корректной
   обработкой permission/согласий (учесть семантику «согласие видно в контексте
   самого вызова», инцидент T-057, и раздел про auto-режим в `knowledge.md`).
   Чисто — идею можно коммитить. Больно — сигнал, что `processor` пока остаётся
   агентом.
2. **Ужесточение контрактов листовых агентов** — параллельно: любой возврат
   coder/reviewer/merger машинно-разбираем без свободного текста. Ценно само по себе,
   независимо от исхода спайка.

## 9. Миграция и cutover

1. После успешного спайка — строить **движок headless** (без TUI), тонким секвенсором
   поверх существующих `tools/*.ps1` (вызов подпроцессами; нативный порт — позже).
2. **`processor.md` держать как исполняемую спецификацию-оракул** на время миграции.
   Вести дифференциальное тестирование: бинарь и проза-процессор на одном сценарии
   обязаны сходиться к одному fingerprint — `tools/harness.ps1` уже вычисляет ровно
   такой timestamp/event-id-независимый отпечаток конечного состояния. Это готовый
   механизм cutover без дрейфа.
3. **TUI — последним**, когда движок уже гоняет когорту сам: обзор + Decision Inbox +
   команды, заменяя `cc-status`/`cc-journal` и отдавая локальный обзор из
   OBSERVABILITY-плана.
4. После стабильного cutover — вывести `processor.md` из эксплуатации; нативный порт
   `tools/*.ps1` — по потребности.

## 10. Выгоды (привязка к признанным болям roadmap)

- **Устранение дрейфа/пропуска шагов**: код не «недочитывает спецификацию»; 243 КБ
  прозы с ежедневным churn — сам по себе сигнал хрупкости.
- **Токены/латентность**: дирижирование перестаёт жечь токены; холостой/no-op цикл
  становится практически бесплатным и мгновенным (roadmap §Этап 2 — прямое попадание).
- **TUI — естественное и уже желаемое**: отдаёт локальный обзор и Decision Inbox из
  `OBSERVABILITY_PLATFORM_PLAN.md §6` без отдельного event-store + web-API сервиса;
  для одномашинного сценария проще запланированной платформы.
- **Тестируемость**: фазовая логика — first-class код с unit/property-тестами вместо
  прозы, проверяемой consistency-скриптами.
- **Readiness-аудит, бюджеты, circuit-breaker** (roadmap §Этап 2, §Этап 7) —
  нативные инварианты, а не «модель, которую попросили не забыть».

## 11. Риски и смягчения

- **R1. Среда спавна/супервизии субагентов — главная недооценённая стоимость.**
  Сегодня `processor.md` работает *внутри* Claude Code и порождает субагентов
  директивой «Use the X subagent», опираясь на permission/classifier-модель и
  сессионные гранты. Движок *вне* Claude Code должен гнать Claude/Codex через
  CLI/подпроцессы и переизобрести обвязку прав/согласий, streaming, `maxTurns`.
  Тонкость: `tools/supervisor.ps1` описан как обёртка «одного дочернего вызова», но в
  прозе процессора вызов выражен как in-process Claude-Code-субагент; **как именно
  `supervisor.ps1` спавнит субагента сегодня — проверить в первую очередь** (§8.1).
  *Смягчение*: спайк до принятия решения.
- **R2. Остаточное суждение переносится на границу.** Где листовой агент вернул
  неожиданное, детерминированный движок может только эскалировать — не «дорассуждает»,
  как LLM. На старте возможен рост эскалаций. *Смягчение*: §8.2 (ужесточение
  контрактов); чистая эскалация приемлема и лучше импровизации.
- **R3. Потеря graceful degradation.** LLM-процессор переваривает новые/пограничные
  состояния суждением; компилируемый цикл обрабатывает ровно закодированные случаи и
  чисто останавливается на прочих. *Смягчение*: предсказуемая чистая эскалация.
- **R4. Два источника истины на время миграции** (`processor.md` vs бинарь). Дрейф —
  ровно тот отказ, с которым проект уже борется (`check-consistency.ps1`).
  *Смягчение*: короткий cutover через fingerprint (§9.2), не долгая двойная жизнь.
- **R5. Сборка/дистрибуция**: появляется build/release-конвейер и матрица платформ.
  *Смягчение*: один статический бинарь (Rust); pwsh-инструменты уже решают
  кросс-платформенность и остаются как есть.

## 12. Отношение к существующим планам

- Идея **приближает** `LOOP_ORCHESTRA_ROADMAP.md §7.6` («если prompt/state-контракты
  станут слишком хрупкими — вынести детерминированные части в CLI»). 243 КБ прозы с
  ежедневным churn — сильная улика, что условие уже наступает. §8.1 предписывает
  «сначала Markdown-контракт и launcher», поэтому старт оформлен как спайк + ядро, а
  не как немедленный продукт.
- Идея **опережает по UX** `OBSERVABILITY_PLATFORM_PLAN.md`: TUI отдаёт локальные
  экраны (обзор, Decision Inbox) на том же событийном контракте (events вверх,
  commands вниз), закрывая большую часть одномашинной потребности до web/Android.

## 13. Не-цели (чего не делать)

- Не переписывать `processor.md` big-bang и не сносить агента до доказанной
  эквивалентности по fingerprint.
- Не переписывать `tools/*.ps1` на старте — оставить и вызывать подпроцессами.
- Не выбирать язык под Agent SDK (не тянуть движок в TS/Python ради SDK).
- Не начинать с TUI — сперва headless-движок.
- Не делать оркестратор «умнее»: он не судит diff и не оценивает качество; это дело
  листовых LLM-агентов.
- Не превращать листовых агентов в детерминированный код.

## 14. Резюме

Компилируемый **Rust** (по умолчанию; C#/.NET — если так быстрее команде) для мозга
и TUI; **PowerShell-скрипты** для рук (уже есть, оставить); **агент `processor.md`** —
только как временный эталон. Старт — де-рискующий спайк среды спавна субагентов +

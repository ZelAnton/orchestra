# План платформы наблюдаемости и коммуникации Orchestra

## 1. Оценка идеи

Идея правильная и логично дополняет Orchestra. Сейчас оркестр умеет выполнять работу, но
оператору приходится смотреть локальные `.work/status.md`, `.work/journal.md` и файлы задач
в каждом репозитории. При нескольких проектах это создаёт новую ручную работу: искать
активные сессии, замечать блокировки, сопоставлять расходы и возвращаться в нужный терминал
для ответа.

Платформа должна решать три задачи:

1. **Наблюдение:** единый актуальный обзор всех репозиториев, задач, запусков и бюджетов.
2. **Решения:** короткий inbox вопросов, где ответ разблокирует только зависимую работу.
3. **Память:** хронология событий и ежедневные отчёты без чтения сырых agent logs.

Главная поправка к первоначальному представлению: веб-сайт и Android-приложение не должны
становиться вторым оркестратором. Они являются control plane — показывают проекции состояния
и принимают команды человека. Выполнение, VCS и переходы задач остаются у repo runner и
`processor`. Иначе появятся два источника истины и трудно воспроизводимые гонки.

## 2. Неблокирующий human in the loop

### Базовое правило

Решение человека блокирует не processor, репозиторий или весь батч, а минимальный набор:

- саму задачу;
- её прямых и транзитивных dependents;
- задачи с конфликтующим write-domain, если продолжение сделает ожидающий diff устаревшим;
- конкретную фазу публикации, если вопрос относится только к publish.

Все независимые задачи остаются кандидатами для следующих батчей. Processor не держит
глобальный lock во время ожидания человека.

### Где ставить gate

| Момент | Что происходит |
|---|---|
| До планирования | Неясная задача получает `blocked_human`; остальные планируются |
| После планирования | Вопрос о scope/risk паркует только эту задачу до создания worktree |
| После verification | Risky-задача сохраняет branch/worktree и ждёт approve **до** join-барьера |
| При merge-конфликте | Задача карантинится и ждёт решения; merger продолжает остальные |
| После CI | Только затронутая публикация/задача блокируется; новые независимые батчи разрешены |

Gate следует размещать до общей интеграции, когда это возможно. Если сначала объединить
несколько задач, а затем запросить одно approval, они искусственно образуют единую блокировку.

### Объект решения

Вопрос человеку должен быть отдельной сущностью, а не сообщением в чате:

```json
{
  "decision_id": "D-20260707-0042",
  "repository_id": "repo-orchestra",
  "task_id": "T-031",
  "type": "approve_publish",
  "question": "Публиковать изменение публичного контракта?",
  "recommended_option": "request_changes",
  "options": ["approve", "request_changes", "cancel"],
  "reason": "Изменены 2 публичных метода",
  "evidence_refs": ["diff:abc..def", "test:run-812"],
  "blocks": ["T-031", "T-044"],
  "status": "pending",
  "created_at": "<ISO-8601>",
  "expires_at": null
}
```

Обязательные свойства:

- один конкретный вопрос;
- рекомендуемый вариант с причиной, но без скрытого автоматического выбора;
- 2–4 осмысленных варианта ответа;
- краткие последствия каждого варианта;
- evidence: diff, проверки, риск, предыдущие попытки;
- явный список заблокированных сущностей;
- автор, время, история изменения и ответивший человек;
- безопасное поведение при отсутствии ответа: обычно продолжать ждать, а не выбирать за
  человека.

### Возобновление

1. UI записывает ответ как команду `decision.resolve`, но не меняет `.work` напрямую.
2. Runner соответствующего репозитория забирает команду.
3. Runner проверяет `decision_id`, допустимость перехода и актуальность branch/SHA.
4. Он применяет ответ идемпотентно и эмитит `decision.resolved` или
   `decision.rejected_as_stale`.
5. Задача переходит в `ready`, `changes_requested` или `cancelled`; dependents
   пересчитываются.
6. Scheduler подхватывает готовую работу в обычном порядке — отдельную долгую сессию держать
   не требуется.

Если за время ожидания base branch изменился, approve не должен молча публиковать старый
результат: задача проходит rebase и повторную verification, после чего решение либо остаётся
применимым по policy, либо запрашивается снова.

## 3. Предложения, задачи и версия назначения

### 3.1. Два типа элементов в едином backlog

Очередь должна принимать не только готовые задачи, но и сырые предложения. Это разные
сущности с разными гарантиями:

- **Proposal** — наблюдение, пожелание, идея или предполагаемая проблема. Оно не разрешает
  реализацию и не обязано содержать правильное решение.
- **Task** — проверенная, достаточно конкретная единица работы с критериями, risk,
  dependencies, conflict domains и версией назначения. Только task может попасть к planner
  и coder.

Единый backlog удобен для обзора и приоритизации, но `kind` должен быть явным. Processor
никогда не должен интерпретировать proposal как task по одному лишь положению в файле или
очереди.

Минимальное представление в очереди:

```markdown
### [P-042] Упростить обновление схемы — kind: proposal — status: proposed
Suggested target: next_major
Source: user
Idea: заменить текущий механизм генерацией на лету.

### [T-118] Добавить версионированный schema cache — kind: task — status: queued
Origin: P-042
Delivery target: current
Rationale: решает подтверждённую проблему без runtime-генерации и breaking changes.
```

Физически это может остаться один `.work/Tasks_Queue.md` на первом этапе. При росте объёма
проекции можно разделить, не меняя доменную модель и идентификаторы.

### 3.2. Opus proposal curator

Отдельный высокоинтеллектуальный проход (`proposal_curator`, Opus/high или выше) превращает
сырой запрос в инженерное решение. Он обязан:

1. Прочитать исходное предложение, релевантный код, constraints, roadmap и knowledge.
2. Проверить, существует ли проблема и не дублирует ли она backlog/готовую работу.
3. Отделить потребность от предложенного автором способа решения.
4. Рассмотреть более простую или более системную альтернативу.
5. Оценить последствия, совместимость, стоимость и подходящую версию.
6. Вернуть один исход:
   - `converted` — создать одну или несколько tasks;
   - `rejected` — проблема не подтверждена или ценность ниже стоимости;
   - `duplicate` — связать с существующим элементом;
   - `needs_human` — требуется продуктовый выбор или неизвестен intent;
   - `deferred` — разумно, но пока нет условий/данных для задачи.

Curator может существенно изменить формулировку и предлагаемое решение, но не должен
переписывать исходный proposal. Исходный текст и автор сохраняются неизменно; рядом хранится
evaluation, rationale, рассмотренные альтернативы и ссылки на созданные task ID. Благодаря
этому видно, какую потребность решает итоговая задача и почему исходный способ был изменён.

Рекомендуемая схема:

```text
P-042 proposed -> evaluating -> converted -> T-118, T-119
                         |----> rejected / duplicate / deferred
                         |----> blocked_human(D-017) -> evaluating
```

Curator не должен становиться дорогим polling-loop. Он запускается для новых proposals
пакетом, по событию или вручную; одинаковые предложения дедуплицируются до Opus-вызова.

### 3.3. Current и next major

Каждая созданная task получает обязательное поле:

```text
delivery_target: current | next_major
```

- `current` — работа допустима в текущей major-линии и обязана соблюдать текущие
  compatibility guarantees.
- `next_major` — intentional breaking change, миграция, удаление legacy-контракта или работа,
  ценность которой раскрывается только в следующей major-версии.

Proposal может иметь лишь авторскую подсказку `suggested_target`; окончательный
`delivery_target` назначает curator. Человек может изменить его отдельным audited decision.

Инварианты:

- обычный processor выбирает только `current`;
- current-task не может зависеть от next-major task;
- next-major task может зависеть от current foundation work;
- обнаруженный breaking impact переводит task из `current` в `next_major` либо требует
  явного human exception;
- next-major backlog видим, дедуплицируется и уточняется, но не конкурирует за обычный
  execution capacity;
- начало новой major-линии — отдельное человеческое решение. После него выполняется
  контролируемый rollover выбранных tasks, а не массовая автоматическая смена поля.

В UI нужны отдельные представления `Proposals`, `Current` и `Next major`, а также связь
proposal → evaluation → task. Ежедневный digest сообщает, сколько предложений принято,
изменено, отклонено и ожидает человека; next-major показывается отдельным компактным блоком,
не создающим ежедневный шум.

## 4. Целевая архитектура без overengineering

```text
Claude/Codex agents
        |
    processor
        | append
 .work/events.jsonl ---- repo runner/outbox ----> API + event store
        ^                                      /       |       \
        | apply command                       Web    Digest   Android/PWA
        +--------- repo runner <---- command inbox <--- decisions
```

### 4.1. Repo side

`processor` продолжает вести привычные Markdown-файлы для человека, но дополнительно пишет
машинные события в `.work/events.jsonl`. JSONL выбран потому, что он append-only, легко
восстанавливается, дружит с Git/файлами и не требует локальной БД.

Небольшой `orchestra-runner` выполняет только четыре функции:

- читает новые события и отправляет их пакетами;
- хранит локальный cursor/outbox при отсутствии сети;
- периодически забирает адресованные репозиторию команды;
- валидирует и передаёт команды processor, после чего подтверждает результат.

Runner не принимает инженерных решений и не редактирует код.

### 4.2. Центральная часть

Для PoC достаточно модульного монолита:

- один HTTP API;
- SQLite для локального PoC, PostgreSQL при постоянном multi-device развёртывании;
- REST для запросов и команд;
- Server-Sent Events или WebSocket только для live-обновлений UI;
- один встроенный background worker для digest и уведомлений.

Kafka, Kubernetes, микросервисы, отдельный time-series stack и сложный workflow engine на
этом этапе не нужны. Event store может быть обычной таблицей append-only событий плюс
материализованные таблицы текущих задач, решений и usage.

### 4.3. Два направления данных

- **Events** идут снизу вверх и описывают уже произошедшее. Они неизменяемы.
- **Commands** идут сверху вниз и выражают намерение человека. Их выполнение подтверждается
  новым событием.

UI никогда не считает команду выполненной сразу после нажатия. Он показывает состояния
`queued → delivered → applied` либо `rejected/stale`. Это устраняет ложное ощущение, что
approve уже повлиял на локальный репозиторий.

## 5. Минимальная модель данных

| Сущность | Назначение |
|---|---|
| `Repository` | Идентификатор, путь/remote, branch, runner health, policy summary |
| `Run` | Один processor/triage запуск, outcome, время, usage, correlation ID |
| `Proposal` | Исходная идея, provenance, evaluation, curator outcome и созданные task IDs |
| `Task` | Состояние, priority, risk, delivery target, dependencies, conflict domains, фаза |
| `Event` | Неизменяемый факт с sequence, type, actor, correlation/causation |
| `Decision` | Вопрос, варианты, blocks, evidence, SLA, ответ и audit trail |
| `Check` | Команда проверки, результат, длительность, ссылка на ограниченный лог |
| `Usage` | Provider/model, actual или estimated tokens/cost, quota snapshot |
| `Notification` | Канал, причина, дедупликационный ключ, delivery/read status |

Минимальный event envelope:

```json
{
  "schema_version": 1,
  "event_id": "<uuid>",
  "repository_id": "repo-x",
  "sequence": 1842,
  "occurred_at": "<ISO-8601>",
  "type": "task.blocked_human",
  "entity_id": "T-031",
  "run_id": "run-812",
  "actor": { "kind": "agent", "name": "processor", "model": "sonnet" },
  "payload": {}
}
```

`event_id` и `(repository_id, sequence)` обеспечивают дедупликацию. Payload версионируется;
старые события никогда не переписываются. Проекции можно полностью восстановить replay-ем.

### Рекомендуемые состояния задачи

```text
queued -> planned -> running -> verifying -> ready_for_integration
  |          |          |          |
  +-------> blocked_human <---------+
                 |
        ready / changes_requested / cancelled

ready_for_integration -> integrating -> publishing -> completed
                                |             |
                           quarantined      failed
```

`blocked_human` — причина, а не терминальный исход. Поле `blocked_by` содержит decision ID
или task ID. Зависимости хранятся явно, чтобы UI и planner могли объяснить каскад блокировки.

## 6. Пользовательский опыт

### 6.1. Главный экран

На одном экране должны помещаться:

- репозитории: running, idle, degraded, offline;
- активные задачи и текущая фаза;
- число решений, требующих человека;
- заблокированные задачи и причина;
- красный CI/ошибки runner;
- расход сегодня и прогноз до конца периода;
- последние завершённые результаты.

Сначала показываются отклонения и действия, а не поток всех событий. Зелёная нормальная
работа сворачивается.

### 6.2. Decision Inbox

Это главный интерфейс human in the loop. Карточка должна отвечать на вопросы:

1. Что от меня требуется?
2. Почему агент не может решить сам?
3. Что рекомендуется и почему?
4. Что будет разблокировано?
5. Какие проверки пройдены?
6. Что изменилось в коде?
7. Насколько срочно отвечать?

Полезные действия: approve, request changes с текстом, выбрать альтернативу, cancel,
отложить/remind later, открыть diff или полный timeline. Для опасных операций нужен явный
confirm; для обычных — одно нажатие.

### 6.3. Экран репозитория

- отдельные lanes `proposals / current / next major` и канбан-проекция состояния tasks;
- dependency graph только по запросу, не как основной экран;
- timeline запуска с группировкой повторяющихся событий;
- очередь и причины приоритета;
- policy, autonomy mode и pause/resume;
- budget/usage по provider/model/role;
- фильтры по task, agent, risk, branch, run.

### 6.4. История задачи

Timeline должен показывать не внутренний chain-of-thought, а проверяемые факты:

- кем и почему создана задача;
- из какого proposal и curator rationale она получена;
- план и выбранный risk;
- коммиты/diff summary;
- reviewer findings и исправления;
- команды тестов и результаты;
- вопросы/решения человека;
- публикацию и CI;
- итог и ссылки на артефакты.

### 6.5. Ежедневный отчёт

Один digest в выбранное время:

- завершено и опубликовано;
- продолжается;
- ждёт человека с возрастом ожидания;
- failed/quarantined и что будет дальше;
- ключевые reviewer findings;
- расход actual/estimated и приближение к budget;
- новые proposals и результаты их Opus-оценки;
- health runner/scheduler;
- 1–3 рекомендации, требующие внимания.

Не отправлять отдельное уведомление о каждом успешном шаге.

## 7. Нотификации и переписка

### Приоритеты

- **Critical:** нарушение safety, возможная потеря данных, неожиданный push, исчерпан hard
  budget. Немедленный push.
- **Action required:** новое решение, stale approve, повторный провал. Push с deep link.
- **Warning:** runner offline, 80% бюджета, растущая очередь. Группировать.
- **Info:** завершение задач. Только digest или подписка пользователя.

Нужны dedupe, quiet hours, per-repository subscriptions и escalation reminders. Повторное
уведомление отправляется не по таймеру вслепую, а если решение всё ещё pending и реально
блокирует работу.

### Переписка

Не создавать универсальный чат на первом этапе. Удобнее тематический thread внутри
`Decision` или `Task`: сообщения имеют автора, timestamp, attachment/evidence и могут
порождать `request_changes`. Это сохраняет контекст и не превращает платформу в ещё один
мессенджер.

Свободный чат можно добавить позже для thinker/планирования, но он не должен подменять
структурированные решения.

## 8. Квоты и бюджетирование LLM

Следует разделять три разных понятия:

1. **Usage:** фактически использованные input/output/cache tokens по вызову.
2. **Cost:** денежная оценка по тарифу модели.
3. **Provider quota:** внешний лимит аккаунта/подписки, который не всегда доступен через API.

Если provider не отдаёт точные usage/quota, интерфейс обязан маркировать данные как
`estimated`, а не смешивать их с actual. Для управления loop важнее внутренние hard limits,
чем попытка идеально угадать внешнюю квоту.

Полезные срезы:

- сегодня/неделя/месяц;
- по репозиторию, pattern, run, task, роли, provider и модели;
- токены на завершённую задачу;
- токены на failed/no-op;
- прогноз до конца дня/месяца;
- soft threshold → report-only, hard threshold → не начинать новые expensive actions.

Бюджет задаётся локальной policy и кэшируется runner-ом: потеря связи с центральным API не
должна отключать hard guard.

## 9. Безопасность и приватность

- По умолчанию отправлять metadata, summaries и ссылки, а не исходный код, prompts и полные
  CI logs.
- Перед отправкой выполнять redaction секретов и ограничивать размер excerpt.
- Не сохранять chain-of-thought; хранить решения, evidence и краткие объяснения.
- Runner аутентифицируется отдельным ключом на репозиторий/машину; ключ можно отозвать.
- Android/web пользователь имеет обычную auth-сессию; опасные решения требуют re-auth или
  дополнительного подтверждения.
- Каждая команда имеет audit trail: кто, когда, с какого устройства, что именно одобрил.
- API не получает произвольный shell-доступ. Допустимые команды перечислены и
  валидируются runner-ом.
- Локальный outbox переживает offline; команды имеют expiry и optimistic concurrency по
  expected task version/SHA.

## 10. Последовательность PoC

### PoC 0. Событийный контракт без UI (1 неделя)

- Зафиксировать lifecycle proposal → curator evaluation → task и правила current/next major.
- Зафиксировать 15–20 основных event types.
- Научить processor писать `.work/events.jsonl` параллельно текущим Markdown-файлам.
- Сделать replay-скрипт, строящий snapshot и проверяющий sequence/idempotency.
- Прогнать на сохранённых сценариях: success, retry, quarantine, crash/resume.

Критерий: snapshot после replay совпадает с фактическим `.work/status.md`.

### PoC 1. Локальный read-only web (1–2 недели)

- Один процесс читает события одного репозитория.
- SQLite и экраны overview, proposals/current/next-major, task timeline, run history.
- Live update через SSE; без auth, команд и облачного deployment.
- Показать active/queued/blocked/done и проверки.

Критерий: для понимания активного батча терминал не нужен.

### PoC 2. Неблокирующие решения (2–3 недели)

- Добавить `Decision`, command inbox и thread.
- Processor паркует одну risky-задачу до integration и продолжает независимую.
- Web позволяет approve/request changes/cancel.
- Runner применяет команду идемпотентно и проверяет stale SHA.

Обязательный demo: T-1 ждёт approve, T-2 выполняется и публикуется, approve T-1 возвращает
её в verification/integration без перезапуска старой сессии.

### PoC 3. Multi-repository hub (2–4 недели)

- Стабильный `repository_id`, runner registration и heartbeat.
- Локальный outbox, batch upload, cursor и offline recovery.
- Единый dashboard и фильтры по нескольким репозиториям.
- Auth для одного владельца; deployment на домашнем сервере/VPS/private network.

Критерий: отключение API не останавливает безопасную локальную работу и не теряет события.

### PoC 4. Уведомления и digest (2 недели)

- Email/Telegram или web push как первый канал — выбрать один.
- Action-required push, quiet hours, dedupe и daily digest.
- Deep links прямо в decision/task.
- Метрики delivery/read/decision latency.

Критерий: нет уведомлений о no-op/success, но каждый pending decision доставлен.

### PoC 5. Usage и бюджеты (2–3 недели)

- Инструментировать вызовы Claude/Codex доступными actual usage.
- Для недоступных данных — versioned estimator.
- Dashboard actual/estimated, soft/hard threshold и прогноз.
- Hard guard остаётся локальным; UI может менять policy только через audited command.

### PoC 6. Mobile-first и Android (2–4 недели +)

Сначала сделать responsive PWA: decision inbox, push, digest, repository health. Это быстро
проверит UX на телефоне и может закрыть большую часть потребности. Нативное Android-приложение
имеет смысл после стабилизации API, если нужны более надёжные push, background sync,
биометрическое подтверждение и лучший просмотр diff.

Для native v1 достаточно четырёх экранов: Home, Decisions, Repository, Task. Не переносить
туда административные настройки и полный log explorer.

### PoC 7. Укрепление (после реального использования)

- backup/restore и event schema migrations;
- retention: summaries надолго, тяжёлые logs ограниченное время;
- role-based access, если появятся другие пользователи;
- SLO runner heartbeat и command delivery;
- собственный health/readiness audit;
- нагрузочное тестирование только по фактическому объёму событий.

## 11. Что не делать в первых версиях

- Не строить визуальный редактор workflow.
- Не давать UI произвольный shell или прямую запись в Git.
- Не хранить весь transcript модели и chain-of-thought.
- Не начинать с native Android до проверки responsive web.
- Не внедрять микросервисы, Kafka, Kubernetes и отдельный data lake.
- Не делать граф зависимостей главным экраном.
- Не превращать все события в push-уведомления.
- Не смешивать estimated tokens с фактической квотой provider.
- Не блокировать весь repo из-за одного pending decision.
- Не позволять coder/planner брать proposal до явного `converted`.
- Не смешивать next-major backlog с текущей execution queue.

## 12. Ближайшие проектные решения

До реализации следует письменно выбрать:

1. Где будет работать центральный сервис: только локальная сеть, private VPS или облако.
2. Какие metadata допустимо отправлять вне рабочей машины.
3. Какой первый канал уведомлений использовать.
4. Как идентифицировать репозиторий после перемещения/клонирования.
5. Как долго хранить diff excerpts и CI logs.
6. Какие три human decision type нужны первыми. Рекомендация: `clarify_scope`,
   `approve_risk`, `approve_publish`.
7. Какой набор событий является достаточным для PoC 0.
8. Где хранится каноническая current major version и как выполняется rollover next-major.

Начинать следует не с дизайна dashboard, а с task/decision semantics и event contract.
Если они устойчивы, web и Android становятся обычными клиентами. Если они расплывчаты,
красивый UI лишь скроет неоднозначное состояние loop.

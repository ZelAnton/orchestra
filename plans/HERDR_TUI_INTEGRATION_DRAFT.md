# Интеграция Orchestra TUI с Herdr — план после критического ревью

> Статус: **переработанный план** (ревизия 3, 2026-07-20). Первая редакция была рабочей
> гипотезой; эта ревизия — результат её критического разбора: проверена фактическая
> документация Herdr v0.7.4, изучен текущий код `tui/` и `engine/`, принят ряд решений,
> отвергнуты избыточные элементы. Оставшиеся неопределённости сведены к короткому списку
> спайков (§10) и вопросов владельцу (§12). До прохождения спайков план не переводится в
> статус утверждённого runtime-контракта. Ревизия 3 вшивает ответы владельца (геометрия,
> масштаб, гибридная модель UI, SSH/Termux). Журнал отличий — в §14.

## 1. Цель

Сделать локальный операторский интерфейс Orchestra пригодным для малых терминальных
ячеек грида (2×4 на монитор, §3) и одновременно расширить его возможности за счёт Herdr:

- основной экран показывает главные события оркестра, активных агентов и команды;
- дополнительные представления живут в отдельных вкладках Herdr и открываются по требованию;
- агрегированное состояние проекта (working/blocked) видно в sidebar Herdr без открытия вкладок;
- закрытие TUI или отсоединение клиента Herdr не останавливает headless engine;
- engine остаётся работоспособным без Herdr и не переносит в UI логику оркестрации;
- вкладки создаются и восстанавливаются программно, идемпотентно и без перехвата фокуса;
- human-in-the-loop, сообщения и команды используют структурированные контракты, а не
  произвольную передачу текста в shell или терминал агента.

## 2. Фактология Herdr (проверено по документации v0.7.4, июль 2026)

Herdr — терминальный workspace-менеджер на Rust (актуальный релиз v0.7.4 от 2026-07-15,
активная разработка). Модель: фоновый server + присоединяемые клиенты; workspace → tabs →
panes; pane — настоящий терминальный процесс, переживающий detach клиента. Управление: CLI и
newline-delimited JSON socket API (Unix socket / named pipe на Windows), `--json` у команд,
`herdr api schema --json`, `herdr status`.

Существенные для нас примитивы:

- `herdr workspace|tab|pane create/list/get/focus/rename/close` — адресуемые сущности;
- **`herdr agent start <name> [--cwd PATH] [--workspace ID] [--tab ID] [--split right|down]
  [--env KEY=VALUE] [--focus|--no-focus] -- <argv...>`** — argv-запуск произвольной команды
  с размещением и без перехвата фокуса. Это штатный путь запуска view-процессов; трюки с
  `tab create` + `pane run` (впечатывание команды в shell) не нужны;
- **`herdr pane report-agent <pane_id> --source ID --agent LABEL
  --state idle|working|blocked|unknown [--message TEXT] [--seq N]`** — самодекларация
  состояния процесса; состояние агрегируется в sidebar workspace'а. Это готовый канал
  «агрегированного статуса Orchestra» без кастомного UI;
- `session.snapshot`, `events.subscribe`/`events.wait` в socket API — начальная синхронизация
  и подписка (для CLI-моста достаточно snapshot + повторного reconcile);
- именованные сессии: `herdr --session <name>` / env `HERDR_SESSION`; сокет на сессию
  (`~/.config/herdr/sessions/<name>/herdr.sock`); сессии — изолированные namespace'ы;
- внутри управляемых процессов доступны `HERDR_ENV`, `HERDR_SOCKET_PATH`, `HERDR_WORKSPACE_ID`,
  `HERDR_TAB_ID`, `HERDR_PANE_ID` — детект и адресация без конфигурации;
- плагины: manifest `herdr-plugin.toml`, entrypoints actions/events/panes; plugin pane — обычный
  pane, запускающий argv-исполняемый файл, с размещениями `overlay|popup|split|tab|zoomed`;
  API плагина = весь CLI. На Windows — preview/best effort;
- chrome настраивается: sidebar сворачивается/скрывается, `ui.*`-ключи конфигурации; префикс
  клавиш по умолчанию `ctrl+b` (обычные клавиши, включая `Tab`, проходят в pane).

Ограничения Windows (experimental beta): ConPTY panes, PowerShell/cmd, локальные persistent
sessions, agent detection работают; плагины — preview; live handoff и direct attach не
поддерживаются; возможны артефакты курсора и различия в modified keys (`shift+enter`);
**процессы pane не переживают полный рестарт Herdr server** — восстанавливается только layout.

Лицензия: AGPL-3.0+ с коммерческой опцией. Orchestra вызывает `herdr` как внешний субпроцесс и
не линкуется с ним — обязательств на код Orchestra это не накладывает.

Источники: <https://herdr.dev/docs/concepts/>, <https://herdr.dev/docs/cli-reference/>,
<https://herdr.dev/docs/socket-api/>, <https://herdr.dev/docs/agents/>,
<https://herdr.dev/docs/plugins/>, <https://herdr.dev/docs/session-state/>,
<https://herdr.dev/docs/preview/windows-beta/>, <https://github.com/ogulcancelik/herdr>.

Вердикт реализуемости: **реализуемо без изменения Herdr**, включая Windows, с оговоркой на
beta-статус Windows-порта. Точные семантики двух команд (`agent start` с произвольным именем,
видимость `report-agent` в sidebar) закрепляются спайками §10.

## 3. Геометрия, топология и удалённый доступ

Фактическая раскладка (ответы владельца, 2026-07-20): один–два монитора, на каждом — грид
**2×4** (8 ячеек; всего до 16, чаще достаточно одного грида) в Windows Terminal; параллельно
живут 4–12 проектов, чаще 6–7. Полноэкранный Windows Terminal даёт порядка 200–320 колонок ×
50–80 строк (зависит от шрифта/монитора); ячейка грида 2×4 — грубо 50–160 колонок × 12–35
строк в зависимости от ориентации сетки. Это диапазон `compact`, местами `full` (§7);
сценарий 20×7 из первой редакции — не основной случай, а деградационный запас и режим узких
SSH-клиентов (§3.1). Точные размеры снимает спайк S4.

Принятая модель UI — **гибрид** (решение владельца):

- **В норме** ячейка минималистична: Overview (лог ключевых событий + активные задачи/агенты +
  статус-строка), chrome Herdr свёрнут (sidebar collapsed), остальные представления — по
  горячим клавишам через компактное меню (D5, §7).
- **По требованию** ячейка разворачивается на весь экран (zoom pane Windows Terminal или
  максимизация окна): TUI сам переходит в `full`-композицию по resize, оператор раскрывает
  sidebar Herdr — workspace проекта, вкладки и агрегированные состояния (`report-agent`)
  становятся полноразмерными.

Топология сессий: **сессия Herdr на проект** (T1). При 4–12 проектах это 4–12 фоновых
серверов Herdr — приемлемо (замер в §9); консолидация нескольких проектов в одну сессию с
общим sidebar (T2) на этом масштабе не требуется и остаётся опцией (S3 — опциональный спайк).

### 3.1. Удалённый доступ (SSH, Termux)

Требование: доступ к тем же интерфейсам по SSH с других машин и с Android (Termux). Пути, от
гарантированного к богатому:

1. **`orchestra-tui` напрямую в SSH-сессии** (standalone, без Herdr): работает тривиально,
   адаптивные режимы §7 покрывают телефонный экран (портрет Termux — примерно 50–80×25–45
   символов); вкладок нет, но внутренние экраны и хоткей-меню есть. Базовый путь.
2. **Клиент Herdr в SSH-сессии** к работающему на Windows-хосте серверу проекта — вкладки и
   sidebar удалённо. Локальный attach на Windows beta работает, но «direct terminal attach» и
   `herdr --remote` числятся в ограничениях; работоспособность attach из SSH-сессии
   (OpenSSH → ConPTY) требует проверки — спайк S7.
3. Долгосрочно тонкие удалённые клиенты (web/Android) закрывает
   `OBSERVABILITY_PLATFORM_PLAN.md`; Herdr-слой его не заменяет и не блокирует.

## 4. Архитектура и границы

Сохраняется инвариант `DETERMINISTIC_ORCHESTRATOR_INTENT.md` §5: engine — headless-мозг,
TUI — проекция и пульт; события вверх, команды вниз; закрытие UI не трогает engine.

```text
                         .work/events.jsonl + snapshot .work/
Orchestra engine ------------------------------------------------> view-процессы (orchestra-tui --view ...)
        ^                                                                 |            |
        | typed command + durable ack (этап 3)                            |            | pane report-agent
        +-----------------------------------------------------------------            v
                                                                               Herdr (sidebar)
orchestra-tui herdr <bootstrap|open|reconcile> ---- herdr CLI (--json) ----> Herdr (workspace/tabs)
```

Жёсткие правила:

1. **Engine никогда не живёт внутри pane Herdr.** На Windows beta процессы pane не переживают
   рестарт server — engine запускается снаружи (launcher/сервис). Из pane допустим только
   *detached*-запуск engine launcher'ом (как это уже делают `cc-*`), чтобы время жизни engine
   не было привязано к pane.
2. **Herdr не супервизирует листовых агентов.** Coder/reviewer/planner/merger остаются
   дочерними процессами супервизора Orchestra; их состояние поступает из событий Orchestra.
   Перенос их в panes сломал бы ownership, timeout/cancellation и containment.
3. **Herdr — не источник истины.** Потеря Herdr → engine продолжает/восстанавливается по
   `.work/` и crash-recovery контракту; вкладки создаются заново мостом. IDs Herdr —
   недолговечные подсказки; reconcile идёт по семантическим label'ам.
4. **Мост не принимает оркестрационных решений.** Только bootstrap/reconcile вкладок,
   `report-agent` и открытие view по запросу.
5. **`pane send-text`/`pane run` не являются командным каналом Orchestra** — ни для запуска
   view (есть `agent start`), ни тем более для команд engine (нет audit/idempotency/stale
   protection).

## 5. Принятые решения

| # | Решение | Обоснование | Риск/оговорка |
|---|---|---|---|
| D1 | Первым делается **адаптивный TUI** (full/compact/micro), без какой-либо зависимости от Herdr | нужен в каждом сценарии (standalone, T1, T2); главный выигрыш площади лежит здесь, а не во вкладках | пороги режимов валидируются S5 |
| D2 | **Один бинарь** `orchestra-tui` с флагом `--view overview\|queue\|inbox\|...`; внутренние экраны и клавиша `Tab` сохраняются во всех режимах | вкладки Herdr — аддитивная навигация, не замена; `Tab` не конфликтует с Herdr (перехватывается только префикс `ctrl+b`) | матрица клавиш Windows Terminal — в S5 |
| D3 | Мост — **подкоманды того же бинаря**: `orchestra-tui herdr bootstrap/open/reconcile`, реализация через **CLI-обёртки** (`herdr ... --json`) | вариант A из первой редакции: минимум кода, лучшая переносимость на Windows beta, диагностируемость; socket-клиент — только при доказанной потребности в live-подписке | subprocess на операцию — приемлемо для редких операций моста |
| D4 | Запуск view-вкладок — **`herdr agent start --no-focus ... -- orchestra-tui --view X --work <path>`** | атомарный argv-запуск с размещением; исключает гонки впечатывания команд в shell | точная семантика `<name>`/размещения — спайк S1 |
| D5 | **Ленивые вкладки**: bootstrap создаёт только Overview; все остальные вкладки, включая `Inbox`, — строго по требованию (хоткей-меню в Overview / `orchestra-tui herdr open <view>`; `:open` — с этапа 3); флаг `--full` для полного набора; авто-создание `Inbox` при первом pending-решении — конфиг-опция, по умолчанию выключена | в grid-сценарии 8 вечных view-процессов × N проектов — необоснованный фон; blocked-состояние проекта видно и без вкладки Inbox через D6 | решение владельца (2026-07-20): строго лениво; пересмотр после замеров §9 |
| D6 | Overview-процесс регулярно вызывает **`pane report-agent`** с агрегированным состоянием проекта (§8) | родная видимость в sidebar/workspace без кастомного UI; закрывает роль badge из первой редакции | видимость в sidebar — спайк S2 |
| D7 | Топология сессий: для T1 — **сессия на проект**; консолидация в одну сессию (T2) на масштабе 4–12 проектов не требуется (S3 — опционально) | изоляция фокуса гарантирована; сокет обнаруживается через `HERDR_SOCKET_PATH`/`HERDR_SESSION` | стоимость N серверов Herdr — замерить в §9 |
| D8 | **Этап 1–2 не требует новых типов событий**: Overview — `events.jsonl`+`status.md`, Inbox и Queue — `engine::state::Snapshot` | текущего контракта достаточно для трёх ключевых view; расширение контракта (§7 первой редакции) отвязано от Herdr-работ | Active-список «по агентам» до этапа 3 строится из task-проекции, без agent lifecycle |
| D9 | Командная строка `:` и typed-команды с ack-событиями — **этап 3**, вместе с command handler engine | в micro-режиме одноклавишные команды (`p`,`u`,`a`,`d`…) дешевле строки ввода; контракт команд (§8 первой редакции: command id, actor, expected state, ack `command.applied/rejected/stale`) сохраняется как цель | до этапа 3 действует существующий keystroke-набор |
| D10 | Reconcile идемпотентен и ведётся **по семантическим label'ам** (`orchestra:<view>`), сохранённые IDs — только подсказка; служебная привязка — в `.work/ui/herdr.json` вне контракта engine | IDs не переживают рестарт server; пользовательские вкладки не трогаются | `.work/ui/` — собственность UI-слоя, engine его не читает |
| D11 | Herdr **необязателен**: детект `HERDR_ENV=1` + `herdr status`; иначе — обычный standalone TUI или headless engine | отсутствие Herdr — не ошибка | — |
| D12 | Плагин Herdr — **этап 4, только упаковка** поверх стабильного standalone-моста (manifest actions/panes) | нативная discoverability; но plugin API — preview, на Windows best effort | не делать плагин единственным runtime |

## 6. Информационная архитектура вкладок

Набор ограниченный и семантический; вкладка-на-задачу/попытку/сообщение — запрещена. Просмотр
одной задачи — временная вкладка `Task` (переиспользуемая), не растущий набор.

| View | Назначение | Источник данных | Этап |
|---|---|---|---|
| `Overview` | фаза, ключевые события, активные задачи, статус | есть: `events.jsonl` + `status.md` | 1 |
| `Inbox` | approvals, эскалации, карантин, blocked | есть: `Snapshot` + approvals (существующий Decision Inbox) | 1 |
| `Queue` | ожидающие задачи, порядок, зависимости | есть: `Snapshot` (очередь уже парсится) | 1–2 |
| `Task` | таймлайн одной задачи по запросу | есть: события + дескриптор | 2 |
| `Active` | точный список агентов, роли, попытки | требует agent lifecycle событий | 3 |
| `Messages` | ключевые входящие/исходящие сообщения | требует `message.recorded` | 3 |
| `Knowledge` | изменения KB за сессию | требует `knowledge.changed` (writer — knowledge_curator) | 3 |
| `Usage` | provider/model/tokens, actual/estimated | `usage.recorded` (уже в контракте, нужна проекция) | 3 |
| `Diagnostics` | supervisor/process/cleanup/CI аномалии | требует `diagnostic.recorded` | 3 |

Правила появления: `Overview` — при bootstrap; все остальные, включая `Inbox`, — лениво, по
явному действию оператора (D5): сигнал «нужно решение» приходит и без вкладки — через
Overview и `report-agent` (D6). Авто-создание `Inbox` при первом pending-решении
(идемпотентное, без фокуса) — выключенная по умолчанию конфиг-опция. Новые факты обновляют
`report-agent`-состояние и label вкладки, но **не перехватывают фокус**; auto-focus запрещён
(возможное единственное исключение — явно сконфигурированный critical safety incident).

## 7. Адаптивный TUI

Режим определяется по `Frame::area()` на каждом рендере (resize без перезапуска) — этим же
покрывается «zoom ячейки на весь экран» из §3. По фактической геометрии рабочие режимы —
`compact` и `full`; `micro` — запас деградации и режим узких SSH/Termux-клиентов. Пороги
предварительные, снимаются спайком S5:

| Режим | Порог (предв.) | Поведение |
|---|---|---|
| `full` | ≥ 80×20 | текущая многопанельная композиция |
| `compact` | ≥ 40×12 | одна колонка, сокращённые label, минимум рамок |
| `micro` | ≥ 20×7 | статус-строка, 1–2 активных строки, одно событие, строка команд/уведомлений |
| `too-small` | меньше | одна строка состояния + подсказка zoom |

Пример micro-Overview:

```text
B12 W2 RUN  A3 !1
T132 review codex
09:41 T131 merged
p/u/a… _
```

В micro: без рамок и постоянной легенды клавиш; предсказуемое сокращение ID/label; отклонения
вытесняют зелёные normal-события; списки прокручиваются, а не обрезаются; уведомления и
результаты команд — в одной сменяемой статус-строке; Unicode-глифы с ASCII fallback.

Во всех режимах Overview имеет компактное хоткей-меню (одна клавиша → popup со списком
представлений): в herdr-режиме выбор открывает/фокусирует вкладку через мост, в standalone —
переключает внутренний экран. Это единый механизм «остальное — по горячим клавишам» из §3.

Текущий рендерер (`tui/src/ui.rs`) построен на процентных Constraint'ах и тестируется на
140×40 — он остаётся `full`-режимом; `compact`/`micro` — отдельные композиции, а не «ужатый»
full. Render-тесты дополняются размерами 60×16, 30×10, 20×7.

## 8. Состояние проекта в sidebar (`report-agent`)

Overview-процесс (он всегда жив в ячейке) на каждом содержательном изменении проекции — но не
чаще раза в 1–2 с — вызывает:

```text
herdr pane report-agent $HERDR_PANE_ID --source orchestra --agent "orc <project>" \
  --state <s> --message "<короткая сводка>" --seq <n>
```

Маппинг состояния (детерминированный, из уже имеющейся проекции):

| Условие (по убыванию приоритета) | `--state` | `--message` (пример) |
|---|---|---|
| pending approvals > 0 или эскалации/карантин > 0 | `blocked` | `2 approvals, 1 esc` |
| активная когорта (фаза ≠ closed) | `working` | `B12 W2 join` |
| нет активной когорты, engine жив | `idle` | `idle, queue 7` |
| проекция недоступна/противоречива | `unknown` | `no events` |

Это даёт glanceable-обзор в T2 и honest-статус ячейки в T1 без единой дополнительной вкладки.
Сбой вызова `herdr` — не ошибка TUI (best-effort, с backoff).

## 9. Процессы и ресурсы

Масштаб (§3): 4–12 проектов, чаще 6–7. Дефолт после D5 — ровно один view-процесс (Overview)
на проект, т.е. обычно 4–12 процессов; каждая открытая по требованию вкладка добавляет один
view, худший случай (полный набор на двух проектах разом) — ~14 сверху. Каждый view — Rust-бинарь, tail append-only
файла раз в tick (250 мс по умолчанию; фоновым view уместен `--tick-ms 1000+`). Отдельная
статья — 4–12 фоновых серверов Herdr (сессия на проект): их RSS/CPU idle включить в замеры.

Перед любым усложнением (projection daemon, thin clients, projection API у engine) — замеры на
реальной топологии: RSS/CPU idle одного view; стоимость 4/8/32 view; стоимость N Herdr
servers при T1; поведение при detach/reattach и рестарте server; латентность
событие→экран и команда→ack. **Daemon не вводить без измеренного узкого места** (решение
первой редакции подтверждается).

## 11. Этапы реализации

- **Этап 0.** Спайки §10 (короткие, выбрасываемый код).
- **Этап 1. Адаптивный TUI.** Режимы full/compact/micro + `--view` + render-тесты малых
  размеров + ASCII fallback. Ценен немедленно и без Herdr (ячейка грида с обычным терминалом).
- **Этап 2. Мост.** `orchestra-tui herdr bootstrap/open/reconcile` (CLI, `--json`), детект
  D11, запуск через `agent start` (D4), `report-agent` (D6), привязка в `.work/ui/herdr.json`
  (D10), вкладки Overview/Inbox/Queue/Task. Ничего в engine не меняется.
- **Этап 3. Контракты.** Расширение событий (agent lifecycle, `message.recorded`,
  `knowledge.changed`, `diagnostic.recorded`, `command.*`) в порядке совместимого выката:
  контракт → tolerant readers → writer allowlist/тесты → emission (порядок первой редакции
  сохраняется; `EventType` — закрытый enum, выкат строго последователен). Командная строка
  `:` поверх typed-команд с durable ack. Вкладки Active/Messages/Knowledge/Usage/Diagnostics.
- **Этап 4 (опционально).** Упаковка плагином Herdr: manifest, actions («open Orchestra
  views»), plugin panes; standalone-мост остаётся полнофункциональным fallback.

Этапы 0–2 не трогают engine и не конкурируют с работами по
`DETERMINISTIC_ORCHESTRATOR_INTENT.md`; этап 3 координируется с планом контрактов engine
(intent §7–8). Приоритет engine-работ этот план не меняет (intent §9.3: «TUI — последним»
относится к движку; данный план — про слой представления поверх уже существующего PoC).

## 12. Открытые вопросы владельцу

Ответы владельца (2026-07-20) вшиты в §3: геометрия — до двух мониторов с гридом 2×4 (чаще
достаточно одного), масштаб — 4–12 проектов (чаще 6–7), модель UI — гибрид «минимальная

Дефолт D5 также решён владельцем: строго ленивый bootstrap (Overview-only), `Inbox` — по
явному действию; авто-создание — выключенная по умолчанию опция. Открытых вопросов,
требующих решения владельца, не осталось; ориентация грида 2×4 и рабочий шрифт отдельного
решения не требуют, если S4 подтвердит попадание ячеек в диапазон `compact`.

## 13. Что не делать

- Не запускать engine внутри pane Herdr; не супервизировать листовых агентов через Herdr.
- Не делать Herdr обязательным для engine; не превращать его в источник истины.
- Не создавать вкладку на каждую задачу/попытку/событие; не перехватывать фокус.
- Не использовать `pane send-text`/`pane run` как канал команд или запуска view.
- Не давать UI произвольный shell; не хранить transcripts/chain-of-thought.
- Не вводить projection daemon и socket-клиент без измеренной потребности.
- Не привязывать первую версию к plugin API (preview) — только этап 4 поверх fallback.
- Не эмитить новые типы событий до прохождения порядка выката этапа 3.

## 14. Отличия от первой редакции

Подтверждено ревью: реализуемость на примитивах Herdr; граница engine/UI; CLI-мост как
вариант A; идемпотентный reconcile по label'ам; запрет супервизии листовых агентов; порядок
совместимого выката событий; список «что не делать».

Исправлено/дополнено:

- найден штатный argv-запуск `herdr agent start ... -- <argv>` — снята основная техническая
  неопределённость запуска вкладок; `tab create`+`pane run` исключены;
- найден `pane report-agent` — агрегированный статус в sidebar получил родной механизм
  (первая редакция предполагала его отсутствие);
- добавлена отсутствовавшая модель топологии grid×сессии (§3, D7) — первая редакция вообще не
  рассматривала, как 32 ячейки сосуществуют;
- «4 вкладки при bootstrap» заменены ленивым дефолтом Overview-only (D5) — процессная
  арифметика grid-сценария делает фикс-набор необоснованным;
- зафиксировано, что этапы 1–2 не требуют новых событий (D8) — Herdr-интеграция отвязана от
  тяжёлой работы над контрактами;
- командная строка `:` сдвинута на этап 3 (D9) — в micro-режиме клавиши дешевле;
- конфликт клавиши `Tab` с Herdr переоценён: Herdr перехватывает только префикс `ctrl+b`,
  внутренние экраны сохраняются везде (D2);
- явно запрещён запуск engine внутри pane (правило §4.1) — следствие ограничения Windows beta;
- отмечена лицензия AGPL-3.0+ Herdr: вызов CLI субпроцессом обязательств не создаёт.

Отвергнуто: динамический lifecycle «вкладка-на-сущность»; plugin как основной runtime;
socket-клиент и projection daemon на старте; расширение событий ради первых вкладок.
